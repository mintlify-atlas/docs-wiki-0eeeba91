ABP's CI is intentionally thin — every workflow is a shell call into the same PowerShell scripts that maintainers run locally (see [/build/build-scripts](/build/build-scripts) and [/build/deploy-scripts](/build/deploy-scripts)). The versioning side is also minimal: one `<Version>` element in `common.props` propagates to ~400 NuGet packages via implicit MSBuild import.

## `.github/workflows/` — workflow files

```text .github/workflows/
angular.yml                # PR build/test/lint of npm/ng-packs
auto-pr.yml                # Auto-merge dev → rel-10.0
build-and-test.yml         # The main .NET PR build
cancel-workflow.yml        # Cancels superseded runs
codeql-analysis.yml        # Security scanning (csharp + javascript)
image-compression.yml      # Compress images committed in PRs
labeler.yml                # Periodic PR labeler
publish-release.yml        # Manual workflow_dispatch to create a release
update-versions.yml        # Bump latest-versions.json after release
```

<Info>
There is **no** release-publishing workflow that runs `_run_all.ps1`. Releases are cut by a maintainer locally, then `publish-release.yml` is a manual lightweight wrapper around `actions/create-release` for the GitHub Release entry, and `update-versions.yml` reacts to the release event to update `latest-versions.json`.
</Info>

## `build-and-test.yml` — the main .NET PR build

This is the workflow that gates every PR that touches C# / Razor / props.

```yaml .github/workflows/build-and-test.yml
name: "build and test"
on:
  push:
    branches:
      - dev
    paths:
      - 'framework/**/*.cs'
      - 'framework/**/*.cshtml'
      - 'framework/**/*.csproj'
      - 'framework/**/*.razor'
      - 'modules/**/*.cs'
      - 'modules/**/*.cshtml'
      - 'modules/**/*.csproj'
      - 'modules/**/*.razor'
      - 'templates/**/*.cs'
      - 'templates/**/*.cshtml'
      - 'templates/**/*.csproj'
      - 'templates/**/*.razor'
      - 'Directory.Build.props'
      - 'Directory.Packages.props'
      - '.github/workflows/build-and-test.yml'

  pull_request:
    paths: # same as push
      - ...
    types:
      - opened
      - synchronize
      - reopened
      - ready_for_review
permissions:
  contents: read

jobs:
  build-test:
    runs-on: ubuntu-22.04
    timeout-minutes: 50
    if: ${{ !github.event.pull_request.draft }}
    steps:
    - uses: jlumbroso/free-disk-space@main
    - uses: PSModule/install-powershell@v1
      with:
        Version: latest
    - uses: actions/checkout@v2
    - uses: actions/setup-dotnet@master
      with:
        dotnet-version: 10.0.x
    - name: Build All
      run: ./build-all.ps1
      working-directory: ./build
      shell: pwsh

    - name: Test All
      run: ./test-all.ps1
      working-directory: ./build
      shell: pwsh

    - name: Codecov
      uses: codecov/codecov-action@v2
```

### Anatomy

<Steps>
  <Step title="Path filter">
    Only triggers on actual code changes. PRs that only touch docs, the `npm/` folder, or YAML never re-run this workflow.
  </Step>
  <Step title="Draft guard">
    `if: ${{ !github.event.pull_request.draft }}` skips draft PRs to save runner minutes.
  </Step>
  <Step title="Free disk space">
    `jlumbroso/free-disk-space@main` clears ~25 GB from the Ubuntu runner (Android SDK, Haskell, .NET cache, dotnet docs). Required — a full build + test set crosses the runner's default 14 GB free.
  </Step>
  <Step title="Install pwsh + .NET 10">
    PowerShell is installed because the runner image's pwsh is sometimes lagging. .NET SDK matches `global.json` (`10.0.x`).
  </Step>
  <Step title="Run the build/test scripts">
    `./build-all.ps1` then `./test-all.ps1` in `./build` — exactly the same scripts a maintainer runs locally. No `-f`, so templates are skipped (~10 min vs ~30 min full).
  </Step>
  <Step title="Codecov">
    Uploads `coverage.cobertura.xml` files produced by Coverlet (auto-injected via `Directory.Build.props`).
  </Step>
</Steps>

## `angular.yml` — the npm/ng-packs build

Parallel workflow for the TypeScript/Angular side. Different path filter, different toolchain.

```yaml .github/workflows/angular.yml
name: 'Angular'
on:
  pull_request:
    paths:
      - 'npm/ng-packs/**/*.ts'
      - 'npm/ng-packs/**/*.html'
      - 'npm/ng-packs/*.json'
      - '!npm/ng-packs/scripts/**'
      - '!npm/ng-packs/packages/schematics/**'
    branches:
      - 'rel-*'
      - 'dev'
    types: [opened, synchronize, reopened, ready_for_review]

jobs:
  build-test-lint:
    if: ${{ !github.event.pull_request.draft }}
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
        with:
          fetch-depth: 0

      - uses: actions/cache@v4
        with:
          path: 'npm/ng-packs/node_modules'
          key: ${{ runner.os }}-${{ hashFiles('npm/ng-packs/yarn.lock') }}

      - uses: actions/cache@v4
        with:
          path: 'templates/app/angular/node_modules'
          key: ${{ runner.os }}-${{ hashFiles('templates/app/angular/yarn.lock') }}

      - name: Install packages
        run: yarn install
        working-directory: npm/ng-packs

      - name: Run lint
        run: yarn affected:lint --base=remotes/origin/${{ github.base_ref }}
        working-directory: npm/ng-packs

      - name: Run build
        run: yarn affected:build --base=remotes/origin/${{ github.base_ref }}
        working-directory: npm/ng-packs

      - name: Run test
        run: yarn affected:test --base=remotes/origin/${{ github.base_ref }}
        working-directory: npm/ng-packs
```

The `yarn affected:*` commands use Nx-style change detection via `fetch-depth: 0` (full git history is needed to diff `origin/<base_ref>`).

The negative path filters (`!npm/ng-packs/scripts/**`, `!npm/ng-packs/packages/schematics/**`) skip CI on PRs that only touch build helpers or the schematics package.

## `codeql-analysis.yml` — security scanning

```yaml .github/workflows/codeql-analysis.yml (excerpt)
name: "CodeQL"

on:
  push:
    branches: [dev, rel-*]
    paths:
      - "abp/**/*.js"
      - "abp/**/*.cs"
      - "abp/**/*.cshtml"
      - "abp/**/*.csproj"
      - "abp/**/*.razor"
  pull_request:
    branches: [dev]
    paths: [ ... same ... ]

jobs:
  analyze:
    if: ${{ !github.event.pull_request.draft }}
    strategy:
      fail-fast: false
      matrix:
        language: ["csharp", "javascript"]

    steps:
      - uses: actions/checkout@v2
        with:
          fetch-depth: 2

      - run: git checkout HEAD^2
        if: ${{ github.event_name == 'pull_request' }}

      - name: Initialize CodeQL
        uses: github/codeql-action/init@v1
        with:
          languages: ${{ matrix.language }}

      - name: Autobuild
        uses: github/codeql-action/autobuild@v1

      - name: Perform CodeQL Analysis
        uses: github/codeql-action/analyze@v1
```

The `abp/**` path prefix matches an older fork layout — note paths in the current repo are `framework/**` etc., but the workflow's path filter is conservative (always-runs on rel branches).

## `cancel-workflow.yml` — keep runners free

```yaml .github/workflows/cancel-workflow.yml
name: cancel-workflow
on: [push]

jobs:
  cancel:
    permissions:
      actions: write
    name: 'Cancel Previous Runs'
    runs-on: ubuntu-latest
    timeout-minutes: 3
    steps:
      - uses: styfle/cancel-workflow-action@0.6.0
        with:
          workflow_id: 10629,1299107,2792859,8268314
          access_token: ${{ github.token }}
```

The `workflow_id`s are GitHub's internal numeric IDs for the build/test/angular workflows. When a contributor force-pushes 5 times in 10 minutes, this kills the older runs.

## `auto-pr.yml` — dev → rel auto-merge

```yaml .github/workflows/auto-pr.yml
name: Merge branch dev with rel-10.0
on:
  push:
    branches:
      - rel-10.0

jobs:
  merge-dev-with-rel-10-0:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
        with:
          ref: dev
      - name: Reset promotion branch
        run: |
          git fetch origin rel-10.0:rel-10.0
          git reset --hard rel-10.0
      - name: Create Pull Request
        uses: peter-evans/create-pull-request@v3
        with:
          branch: auto-merge/rel-10-0/${{github.run_number}}
          title: Merge branch dev with rel-10.0
          reviewers: maliming
          draft: true
          token: ${{ github.token }}
      - name: Merge Pull Request
        env:
          GH_TOKEN: ${{ secrets.BOT_SECRET }}
        run: |
          gh pr ready
          gh pr review auto-merge/rel-10-0/${{github.run_number}} --approve
          gh pr merge auto-merge/rel-10-0/${{github.run_number}} --merge --auto --delete-branch
```

Whenever a hotfix is pushed to `rel-10.0`, this fires the reverse merge into `dev` so the two don't drift. The branch name template `auto-merge/rel-10-0/<run-number>` keeps the PRs uniquely named.

## `publish-release.yml` — GitHub Release on demand

```yaml .github/workflows/publish-release.yml (excerpt)
name: Create Release

on:
  workflow_dispatch:
    inputs:
      tag_name:
        description: 'Tag Name'
        required: true
      prerelease:
        description: 'Pre-release?'
        required: true
      branchName:
        description: 'Branch Name'
        required: true

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v2
        with:
          ref: ${{ github.event.inputs.branchName }}

      - name: Create Release
        id: create_release
        uses: actions/create-release@v1
        env:
          GITHUB_TOKEN: ${{ secrets.RELEASE_TOKEN }}
        with:
          tag_name: ${{ github.event.inputs.tag_name }}
          release_name: ${{ github.event.inputs.tag_name }}
          draft: false
          prerelease: ${{ github.event.inputs.prerelease }}

      - name: Checkout code at tag
        uses: actions/checkout@v2
        with:
          ref: ${{ github.event.inputs.tag_name }}

      - name: Build Project
        run: |
          echo "Build project here"
```

This is an *alternative* to `deploy/7-publish-github-release.ps1` — the maintainer can either call the local PS script during `_run_all.ps1`, or after the artifacts are pushed they can dispatch this workflow from the Actions tab to create the release. Both paths POST to the same REST endpoint.

## `update-versions.yml` — keep `latest-versions.json` current

```yaml .github/workflows/update-versions.yml (excerpt)
name: Update Latest Versions

on:
  release:
    types:
      - published

permissions:
  contents: write
  pull-requests: write

jobs:
  update-versions:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - uses: actions/setup-python@v2
        with:
          python-version: 3.x
      - name: Install dependencies
        run: |
          python -m pip install --upgrade pip
          pip install PyGithub
      - name: Update latest-versions.json and create PR
        env:
          GITHUB_TOKEN: ${{ secrets.RELEASE_TOKEN }}
        run: |
          python .github/scripts/update_versions.py
```

Triggered by `release.types: [published]` (i.e. when step 7 of `_run_all.ps1` or `publish-release.yml` succeeds). The Python script reads the new release tag, prepends an entry to `latest-versions.json`, then opens a PR.

### `latest-versions.json` shape

```json latest-versions.json (excerpt)
[
  {
    "version": "9.3.4",
    "releaseDate": "",
    "type": "stable",
    "message": "",
    "leptonx": {
      "version": "4.3.4"
    }
  },
  ...
]
```

Consumers (the ABP CLI, the abp.io website, docs.abp.io) read this file to know what versions exist and which is current.

## Versioning conventions

### `common.props` — the single version

```xml common.props
<Project>
  <PropertyGroup>
    <LangVersion>latest</LangVersion>
    <Version>10.0.1</Version>
    <LeptonXVersion>5.0.1</LeptonXVersion>
    <NoWarn>$(NoWarn);CS1591;CS0436</NoWarn>
    <PackageIconUrl>https://abp.io/assets/abp_nupkg.png</PackageIconUrl>
    <PackageProjectUrl>https://abp.io/</PackageProjectUrl>
    <PackageLicenseExpression>LGPL-3.0-only</PackageLicenseExpression>
    <RepositoryType>git</RepositoryType>
    <RepositoryUrl>https://github.com/abpframework/abp/</RepositoryUrl>
    <PackageReadmeFile>NuGet.md</PackageReadmeFile>
    <PackageTags>aspnetcore boilerplate framework web best-practices angular maui blazor mvc csharp webapp</PackageTags>
    <GenerateDocumentationFile>true</GenerateDocumentationFile>
    <AllowedOutputExtensionsInPackageBuildOutputFolder>$(AllowedOutputExtensionsInPackageBuildOutputFolder);.pdb</AllowedOutputExtensionsInPackageBuildOutputFolder>
  </PropertyGroup>
  <ItemGroup>
    <None Include="..\..\NuGet.md" Pack="true" PackagePath="\"/>
  </ItemGroup>
  <ItemGroup>
    <PackageReference Include="Microsoft.SourceLink.GitHub">
      <PrivateAssets>all</PrivateAssets>
      <IncludeAssets>runtime; build; native; contentfiles; analyzers</IncludeAssets>
    </PackageReference>
  </ItemGroup>
  <ItemGroup Condition="'$(UsingMicrosoftNETSdkWeb)' != 'true' AND '$(UsingMicrosoftNETSdkRazor)' != 'true'">
    <None Remove="*.abppkg.analyze.json" />
    <Content Include="*.abppkg.analyze.json">
        <Pack>true</Pack>
        <PackagePath>content\</PackagePath>
    </Content>
  </ItemGroup>
  <ItemGroup>
    <None Remove="*.abppkg" />
    <Content Include="*.abppkg">
        <Pack>true</Pack>
        <PackagePath>content\</PackagePath>
    </Content>
  </ItemGroup>
  <ItemGroup Condition="$(AssemblyName.EndsWith('HttpApi.Client'))">
    <EmbeddedResource Include="**\*generate-proxy.json" />
    <Content Remove="**\*generate-proxy.json" />
  </ItemGroup>
</Project>
```

### What `common.props` does

| Element                                       | Effect                                                                            |
| --------------------------------------------- | --------------------------------------------------------------------------------- |
| `<Version>10.0.1</Version>`                   | Stamped on every NuGet package and assembly.                                      |
| `<LeptonXVersion>5.0.1</LeptonXVersion>`      | LeptonX UI theme version, consumed by `npm/publish-ng.ps1` via `update-lepton-x-versions`. |
| `<LangVersion>latest</LangVersion>`           | Lets the framework use the newest C# features available in the pinned SDK.        |
| `<NoWarn>CS1591;CS0436</NoWarn>`              | Suppresses "missing XML doc" (CS1591) and "type conflict" (CS0436) at the framework level. |
| `<PackageLicenseExpression>LGPL-3.0-only</PackageLicenseExpression>` | License metadata on the `.nupkg`.                          |
| `<PackageReadmeFile>NuGet.md</PackageReadmeFile>` + the `<None Include="..\..\NuGet.md" />` | Ships the repo-root `NuGet.md` as the package README.       |
| `Microsoft.SourceLink.GitHub`                 | Bakes the GitHub commit SHA into the `.pdb`s for IDE step-into-source.            |
| `AllowedOutputExtensionsInPackageBuildOutputFolder` adds `.pdb` | Includes symbols inside the `.nupkg`, no separate `.snupkg`. |
| `<Content Include="*.abppkg" Pack="true" />` | Ships ABP module metadata files alongside binaries.                               |
| `EmbeddedResource Include="**\*generate-proxy.json"` for `*.HttpApi.Client` | Lets clients self-describe for proxy generation.            |

`common.props` is **imported manually** by each project (`<Import Project="..\..\common.props" />` near the top of every `.csproj` in `framework/`, `modules/`, `templates/`). It's not picked up automatically by SDK convention.

### `Directory.Build.props` — automatic per-project additions

```xml Directory.Build.props
<Project>
  <PropertyGroup>
    <IsTestProject Condition="$(MSBuildProjectFullPath.Contains('test'))
      and ($(MSBuildProjectName.EndsWith('.Tests'))
      or $(MSBuildProjectName.EndsWith('.TestBase')))">true</IsTestProject>
  </PropertyGroup>

  <ItemGroup>
    <PackageReference Condition="'$(IsTestProject)' == 'true'" Include="coverlet.collector">
      <Version Condition="$(MSBuildProjectFullPath.Contains('templates'))">6.0.4</Version>
      <PrivateAssets>all</PrivateAssets>
      <IncludeAssets>runtime; build; native; contentfiles; analyzers</IncludeAssets>
    </PackageReference>
  </ItemGroup>
</Project>
```

Unlike `common.props`, `Directory.Build.props` is **auto-imported** by MSBuild — every `.csproj` under the repo root picks it up implicitly. Its job is narrow:

- Detect test projects via path + naming conventions.
- Auto-add `coverlet.collector` to those projects so `dotnet test --collect:"XPlat Code Coverage"` works without each test project having to opt in.
- Pin Coverlet to `6.0.4` specifically for projects under `templates/` (the templates use a different Coverlet version than the framework tests).

### `Directory.Packages.props` — Central Package Management

```xml Directory.Packages.props (excerpt)
<Project>
  <PropertyGroup>
    <ManagePackageVersionsCentrally>true</ManagePackageVersionsCentrally>
    <CentralPackageFloatingVersionsEnabled>true</CentralPackageFloatingVersionsEnabled>
  </PropertyGroup>
  <ItemGroup>
    <PackageVersion Include="AlibabaCloud.SDK.Dysmsapi20170525" Version="4.0.0" />
    <PackageVersion Include="aliyun-net-sdk-sts" Version="3.1.3" />
    <PackageVersion Include="Aliyun.OSS.SDK.NetCore" Version="2.14.1" />
    <PackageVersion Include="AsyncKeyedLock" Version="7.1.6" />
    <PackageVersion Include="Autofac" Version="8.4.0" />
    <PackageVersion Include="Autofac.Extensions.DependencyInjection" Version="10.0.0" />
    <PackageVersion Include="Autofac.Extras.DynamicProxy" Version="7.1.0" />
    <PackageVersion Include="AutoMapper" Version="14.0.0" />
    <PackageVersion Include="Asp.Versioning.Mvc" Version="8.1.0" />
    <PackageVersion Include="AWSSDK.S3" Version="4.0.7.2" />
    <PackageVersion Include="Azure.Storage.Blobs" Version="12.25.0" />
    <PackageVersion Include="Blazorise" Version="1.8.6" />
    ...
  </ItemGroup>
</Project>
```

CPM is a .NET 8+ feature. With `ManagePackageVersionsCentrally=true`:

- Every `<PackageReference>` in a `.csproj` **must not** specify a `Version`.
- The version is resolved from the matching `<PackageVersion>` in this file.
- Bumping a third-party dependency = one PR that edits this single file.

`CentralPackageFloatingVersionsEnabled=true` allows floating versions like `8.0.*` in this file (used sparingly for Microsoft-prefixed runtime packages).

```xml common.test.props
<Project>

    <NoWarn>$(NoWarn);CS1591;CS0436</NoWarn>

Imported manually by test `.csproj`s. Suppresses doc-comment warnings, turns on runtime config generation, and disables assembly attributes that fight with shared-base-class tests.

```

## Editing `common.props` safely

<Steps>
  <Step title="Don't edit Version by hand for a release">
    Let `deploy/1-fetch-and-build.ps1` do it via XML DOM. That keeps formatting consistent.
  </Step>
  <Step title="Do edit LeptonXVersion when bumping the theme">
    Pair with `--leptonXVersion` parameter when running `publish-ng.ps1`.
  </Step>
  <Step title="Don't add new NoWarn codes without buy-in">
    `NoWarn` is intentionally minimal — adding more silently swallows analyzer feedback.
  </Step>
  <Step title="Do test pack output after touching common.props">
    `cd nupkg && powershell -File pack.ps1` against a single project verifies `.nupkg` metadata.
  </Step>
</Steps>

## Recap

- Nine workflows in `.github/workflows/`, most of them small shell-call wrappers around the same PowerShell scripts maintainers run locally.
- `build-and-test.yml` is the main gate; it calls `build/build-all.ps1` + `build/test-all.ps1`.
- `angular.yml` handles the npm side with `yarn affected:*`.
- `publish-release.yml` is a manual `workflow_dispatch` alternative to `deploy/7-publish-github-release.ps1`.
- `update-versions.yml` listens for the published-release event and bumps `latest-versions.json`.
- One `<Version>` element in `common.props` propagates everywhere; `Directory.Packages.props` centralises third-party versions via .NET 8 CPM; `Directory.Build.props` auto-injects Coverlet into test projects.
