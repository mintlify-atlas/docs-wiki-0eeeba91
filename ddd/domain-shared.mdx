The `Volo.Abp.Ddd.Domain.Shared` package, defined under `framework/src/Volo.Abp.Ddd.Domain.Shared/`, is the smallest and most permissive DDD layer in ABP. Its single purpose is to host symbols â€” enums, constants, error codes, localization keys, and Entity Transfer Objects (ETOs) for distributed events â€” that **both** the server-side `Domain` layer and **client-side** projects (Blazor WASM, mobile, JavaScript SDKs through generated DTOs) need to share. Because clients reach this assembly transitively through `Application.Contracts`, anything you put here must be safe to ship to the browser: no `DbContext`, no `IRepository`, no `HttpClient`, no platform-specific dependencies.

This page walks the package contents file by file, explains the rule "what belongs in `Domain.Shared`", and shows a worked example from the Identity module where the team stored error codes, constants, and ETOs.

There are **no entities, no repositories, no domain services** in this package â€” those live one layer up in [Volo.Abp.Ddd.Domain](/ddd/domain).

## The module class

`AbpDddDomainSharedModule` (`framework/src/Volo.Abp.Ddd.Domain.Shared/Volo/Abp/Domain/AbpDddDomainSharedModule.cs`) is intentionally trivial â€” it just chains the two abstraction modules every DDD module ultimately needs:

```csharp
// framework/src/Volo.Abp.Ddd.Domain.Shared/Volo/Abp/Domain/AbpDddDomainSharedModule.cs
using Volo.Abp.EventBus.Abstractions;
using Volo.Abp.Modularity;
using Volo.Abp.MultiTenancy;

namespace Volo.Abp.Domain;

[DependsOn(
    typeof(AbpMultiTenancyAbstractionsModule),
    typeof(AbpEventBusAbstractionsModule)
)]
public class AbpDddDomainSharedModule : AbpModule
{
}
```

Because this module only depends on **abstraction** modules (`AbpMultiTenancyAbstractionsModule`, `AbpEventBusAbstractionsModule`), it stays shippable to clients without dragging server-side providers along. See the [Modularity](/core/modularity) page for the broader rules around `AbpModule` and `[DependsOn]`.

## What belongs in `Domain.Shared`

Use the following decision matrix when you create a new symbol in your own solution:

| Kind of symbol | Belongs in `Domain.Shared`? | Why |
| --- | --- | --- |
| `enum` (e.g. status, role, claim type) | âś… | Stable, value-only, safely consumed by clients. |
| `static class XxxConsts` (max lengths, defaults) | âś… | Same constants used by entity attributes and DTO validators. |
| `static class XxxErrorCodes` | âś… | Clients need to translate error codes to messages. |
| Localization resources / JSON files | âś… | Loaded by the `IStringLocalizer` factories on both sides. |
| Distributed event ETOs (`EntityCreatedEto`, custom `UserNameChangedEto`) | âś… | Subscribers in other microservices need the types. |
| Entity (`AggregateRoot`) | âťŚ | Behaviour and persistence belong in `Domain`. |
| `IRepository` interfaces | âťŚ | Repositories are server-side abstractions. |
| `ApplicationService` / DTO request types | âťŚ | DTOs go in `Application.Contracts` (see [Application.Contracts](/ddd/application-contracts)). |

## The distributed entity ETOs

Distributed event ETOs let one microservice publish "this entity changed" without leaking the full entity to subscribers. They sit in `framework/src/Volo.Abp.Ddd.Domain.Shared/Volo/Abp/Domain/Entities/Events/Distributed/`.

### IEntityEto and EntityEto

```csharp
// framework/src/Volo.Abp.Ddd.Domain.Shared/Volo/Abp/Domain/Entities/Events/Distributed/IEntityEto.cs
public interface IEntityEto
{
}

// EntityEto.cs
public class EntityEto : IEntityEto
{
    public string EntityType { get; set; }
    public string KeysAsString { get; set; }
    public EntityEto(string entityType, string keysAsString) { ... }
}
```

`EntityEto` carries just enough metadata (entity type name + serialized keys) for receivers to look up the entity in their own database.

### Lifecycle wrappers

`EntityCreatedEto<TEntityEto>`, `EntityUpdatedEto<TEntityEto>`, and `EntityDeletedEto<TEntityEto>` wrap any ETO so subscribers can fan out per-lifecycle:

```csharp
// framework/src/Volo.Abp.Ddd.Domain.Shared/Volo/Abp/Domain/Entities/Events/Distributed/EntityCreatedEto.cs
[Serializable]
[EventName("abp.entity.created")]
[GenericEventName(Prefix = "abp.entity.created.")]
public class EntityCreatedEto<TEntityEto>
{
    public TEntityEto Entity { get; set; }

    public EntityCreatedEto(TEntityEto entity)
    {
        Entity = entity;
    }
}
```

`Updated` and `Deleted` follow the same shape. The `[EventName]` attribute is what ABP's distributed event bus uses to route messages, see the parent [Domain](/ddd/domain) page for the in-process counterparts (`EntityCreatedEventData<TEntity>` etc.).

### IEntityToEtoMapper

Server-side modules implement `IEntityToEtoMapper` to convert their `Entity` instances into ETOs before publishing. The contract lives in `Domain.Shared` so clients also expose the seam:

```csharp
public interface IEntityToEtoMapper
{
    object? Map(object entityObj);
}
```

The actual mapping implementation is registered server-side and uses `EtoMappingDictionary` (also in this folder) to keep a `entityType -> etoType` lookup.

### AbpDistributedEntityEventOptions

`AbpDistributedEntityEventOptions` is the configuration class used in `services.Configure<AbpDistributedEntityEventOptions>(...)`. It exposes:

- `EtoMappings`: a `EtoMappingDictionary` to register entity-to-ETO bindings.
- `AutoEventSelectors`: an `IAutoEntityDistributedEventSelectorList` to opt-in/out specific aggregate roots for automatic publishing.

Because the options class itself lives in `Domain.Shared`, both server and client can talk about the same shape â€” only the registration of values happens server-side.

## When `Domain.Shared` references should worry you

Because the package transitively flows to all clients, accidentally pulling in a heavy dependency will balloon every client SDK. Guard rails:

- Never `using` an EF Core type, MongoDB type, or ASP.NET Core type in a `Domain.Shared` file.
- Never store a service or behaviour â€” keep classes pure data.
- Localization resources are allowed because the `IStringLocalizer` they target ships as part of the abstraction modules already referenced.
- Use embedded virtual file system declarations only when you ship JSON localization next to the module class, exactly as the Identity module does in its `Localization/Resources/AbpIdentity/` folder.

## Hosting a localization resource

If you need to attach a localization resource to your `Domain.Shared` module, the pattern is:

```csharp
public override void ConfigureServices(ServiceConfigurationContext context)
{
    Configure<AbpVirtualFileSystemOptions>(options =>
    {
        options.FileSets.AddEmbedded<AbpIdentityDomainSharedModule>();
    });

    Configure<AbpLocalizationOptions>(options =>
    {
        options.Resources
            .Add<IdentityResource>("en")
            .AddVirtualJson("/Localization/Resources/AbpIdentity");
    });
}
```

The same pattern appears in `framework/src/Volo.Abp.Ddd.Application.Contracts/Volo/Abp/Application/AbpDddApplicationContractsModule.cs`, the next layer up â€” see [Application.Contracts](/ddd/application-contracts).
