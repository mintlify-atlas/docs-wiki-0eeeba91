The `Volo.Abp.Ddd.Application.Contracts` package, defined under `framework/src/Volo.Abp.Ddd.Application.Contracts/`, defines the **client-facing contract surface** of an ABP module. It only ships interfaces (`IApplicationService`, `IReadOnlyAppService`, `ICrudAppService`) and DTOs — never the concrete services. Every client project (Blazor WASM, .NET MAUI, mobile SDK proxies generated by `Volo.Abp.Http.Client`, MVC pages) references this package directly, while the server-side `Application` project ([Application](/ddd/application)) is the only place that implements the interfaces.

This page documents the module wiring, the DTO hierarchy in `Application/Dtos/`, and the `IApplicationService`/CRUD interface family in `Application/Services/`.

## AbpDddApplicationContractsModule

The module class registers the package's embedded JSON localization resource and is dependency-light on purpose:

```csharp
// framework/src/Volo.Abp.Ddd.Application.Contracts/Volo/Abp/Application/AbpDddApplicationContractsModule.cs
using Volo.Abp.Application.Localization.Resources.AbpDdd;
using Volo.Abp.Auditing;
using Volo.Abp.Data;
using Volo.Abp.Localization;
using Volo.Abp.Modularity;
using Volo.Abp.VirtualFileSystem;

namespace Volo.Abp.Application;

[DependsOn(
    typeof(AbpLocalizationModule),
    typeof(AbpAuditingContractsModule),
    typeof(AbpDataModule)
    )]
public class AbpDddApplicationContractsModule : AbpModule
{
    public override void ConfigureServices(ServiceConfigurationContext context)
    {
        Configure<AbpVirtualFileSystemOptions>(options =>
        {
            options.FileSets.AddEmbedded<AbpDddApplicationContractsModule>();
        });

        Configure<AbpLocalizationOptions>(options =>
        {
            options.Resources
                .Add<AbpDddApplicationContractsResource>("en")
                .AddVirtualJson("/Volo/Abp/Application/Localization/Resources/AbpDdd");
        });
    }
}
```

Two important traits:

- It depends on `AbpAuditingContractsModule`, not `AbpAuditingModule`. The "Contracts" variant is the abstraction-only version safe for clients — same pattern as [Domain.Shared](/ddd/domain-shared).
- It depends on `AbpDataModule` because DTOs include `ExtraProperties` via `IHasExtraProperties` (see [Object Extending](/ddd/object-extending)).

## The Application Service marker

```csharp
// framework/src/Volo.Abp.Ddd.Application.Contracts/Volo/Abp/Application/Services/IApplicationService.cs
namespace Volo.Abp.Application.Services;

public interface IApplicationService : IRemoteService
{
}
```

`IApplicationService` extends `IRemoteService` — that one fact is the reason ABP's dynamic HTTP client proxies (`Volo.Abp.Http.Client.DynamicProxying`) can generate typed clients automatically. The convention used everywhere in the framework:

- Every contract interface lives in `Application.Contracts` and extends `IApplicationService`.
- Every concrete implementation lives in `Application` and inherits from `ApplicationService` (which itself implements `IApplicationService`).
- The HTTP host module exposes `IApplicationService` implementations as MVC controllers via `Volo.Abp.AspNetCore.Mvc`.
- The HTTP client side generates proxies for the same interfaces — so a Blazor WASM project only ever references `Application.Contracts`.

`IRemoteService` lives in `framework/src/Volo.Abp.Core/Volo/Abp/IRemoteService.cs` (namespace `Volo.Abp`); the parent `AbpDddApplicationModule` excludes it from the API description so swagger/openapi generation doesn't try to model it as a parameter — see the snippet in [Application](/ddd/application).

## CRUD-style service contracts

The CRUD interfaces compose `IReadOnlyAppService`, `ICreateAppService`, `IUpdateAppService`, `IDeleteAppService`, and intermediate `ICreateUpdateAppService` into a tower of generic overloads. The complete set:

```csharp
// framework/src/Volo.Abp.Ddd.Application.Contracts/Volo/Abp/Application/Services/IReadOnlyAppService.cs
public interface IReadOnlyAppService<TEntityDto, in TKey>
    : IReadOnlyAppService<TEntityDto, TEntityDto, TKey, PagedAndSortedResultRequestDto> { }

public interface IReadOnlyAppService<TEntityDto, in TKey, in TGetListInput>
    : IReadOnlyAppService<TEntityDto, TEntityDto, TKey, TGetListInput> { }

public interface IReadOnlyAppService<TGetOutputDto, TGetListOutputDto, in TKey, in TGetListInput>
    : IApplicationService
{
    Task<TGetOutputDto> GetAsync(TKey id);
    Task<PagedResultDto<TGetListOutputDto>> GetListAsync(TGetListInput input);
}
```

```csharp
// framework/src/Volo.Abp.Ddd.Application.Contracts/Volo/Abp/Application/Services/ICrudAppService.cs
public interface ICrudAppService<TEntityDto, in TKey>
    : ICrudAppService<TEntityDto, TKey, PagedAndSortedResultRequestDto> { }

public interface ICrudAppService<TEntityDto, in TKey, in TGetListInput>
    : ICrudAppService<TEntityDto, TKey, TGetListInput, TEntityDto> { }

public interface ICrudAppService<TEntityDto, in TKey, in TGetListInput, in TCreateInput>
    : ICrudAppService<TEntityDto, TKey, TGetListInput, TCreateInput, TCreateInput> { }

public interface ICrudAppService<TEntityDto, in TKey, in TGetListInput, in TCreateInput, in TUpdateInput>
    : ICrudAppService<TEntityDto, TEntityDto, TKey, TGetListInput, TCreateInput, TUpdateInput> { }

public interface ICrudAppService<TGetOutputDto, TGetListOutputDto, in TKey, in TGetListInput, in TCreateInput, in TUpdateInput>
    : IReadOnlyAppService<TGetOutputDto, TGetListOutputDto, TKey, TGetListInput>,
      ICreateUpdateAppService<TGetOutputDto, TKey, TCreateInput, TUpdateInput>,
      IDeleteAppService<TKey> { }
```

The cascading overloads exist so most consumers can write the short version (`ICrudAppService<UserDto, Guid>`) while still being able to opt into customised inputs/outputs (`ICrudAppService<UserDto, Guid, GetUserListInput, CreateUserDto, UpdateUserDto>`). The matching implementations live in [Application](/ddd/application#crud-app-service-overloads).

### Composed CRUD building blocks

Each of the small pieces is also a usable contract on its own:

```csharp
// ICreateAppService.cs
public interface ICreateAppService<TEntityDto>
    : ICreateAppService<TEntityDto, TEntityDto> { }

public interface ICreateAppService<TGetOutputDto, in TCreateInput>
    : IApplicationService
{
    Task<TGetOutputDto> CreateAsync(TCreateInput input);
}

// IUpdateAppService.cs
public interface IUpdateAppService<TEntityDto, in TKey>
    : IUpdateAppService<TEntityDto, TKey, TEntityDto> { }

public interface IUpdateAppService<TGetOutputDto, in TKey, in TUpdateInput>
    : IApplicationService
{
    Task<TGetOutputDto> UpdateAsync(TKey id, TUpdateInput input);
}

// IDeleteAppService.cs
public interface IDeleteAppService<in TKey> : IApplicationService
{
    Task DeleteAsync(TKey id);
}

// ICreateUpdateAppService.cs
public interface ICreateUpdateAppService<TEntityDto, in TKey>
    : ICreateUpdateAppService<TEntityDto, TKey, TEntityDto> { }

public interface ICreateUpdateAppService<TEntityDto, in TKey, in TCreateUpdateInput>
    : ICreateUpdateAppService<TEntityDto, TKey, TCreateUpdateInput, TCreateUpdateInput> { }

public interface ICreateUpdateAppService<TGetOutputDto, in TKey, in TCreateUpdateInput, in TUpdateInput>
    : ICreateAppService<TGetOutputDto, TCreateUpdateInput>,
      IUpdateAppService<TGetOutputDto, TKey, TUpdateInput> { }
```

If you only need a "create + list" service, you can inherit `IReadOnlyAppService` and `ICreateAppService` directly and skip `ICrudAppService`.

## DTO families

ABP ships two parallel DTO hierarchies: a plain one and an **extensible** one. The plain hierarchy is fine for purely-internal DTOs; the extensible hierarchy inherits from [`ExtensibleObject`](/ddd/object-extending#extensibleobject) so module consumers can add extra fields without forking source.

### IEntityDto and EntityDto

```csharp
// framework/src/Volo.Abp.Ddd.Application.Contracts/Volo/Abp/Application/Dtos/IEntityDto.cs
public interface IEntityDto { }
public interface IEntityDto<TKey> : IEntityDto { TKey Id { get; set; } }

// EntityDto.cs
[Serializable]
public abstract class EntityDto : IEntityDto
{
    public override string ToString() => $"[DTO: {GetType().Name}]";
}

[Serializable]
public abstract class EntityDto<TKey> : EntityDto, IEntityDto<TKey>
{
    public TKey Id { get; set; } = default!;
    public override string ToString() => $"[DTO: {GetType().Name}] Id = {Id}";
}
```

`EntityDto` itself is marked `//TODO: Consider to delete this class` in source — most modules prefer `ExtensibleEntityDto<TKey>` because it gives them the `ExtraProperties` dictionary for free.

### ExtensibleEntityDto

```csharp
// framework/src/Volo.Abp.Ddd.Application.Contracts/Volo/Abp/Application/Dtos/ExtensibleEntityDto.cs
[Serializable]
public abstract class ExtensibleEntityDto<TKey> : ExtensibleObject, IEntityDto<TKey>
{
    public TKey Id { get; set; } = default!;

    protected ExtensibleEntityDto() : this(true) { }

    protected ExtensibleEntityDto(bool setDefaultsForExtraProperties)
        : base(setDefaultsForExtraProperties) { }

    public override string ToString() => $"[DTO: {GetType().Name}] Id = {Id}";
}
```

Because `ExtensibleObject` implements `IHasExtraProperties` and `IValidatableObject`, every DTO that derives from `ExtensibleEntityDto` automatically:

- Carries an `ExtraProperties` dictionary that maps `ObjectExtensionManager` definitions.
- Participates in `ExtensibleObjectValidator.GetValidationErrors` during model binding.

### Auditing variants

The auditing DTOs add the same `CreationTime`, `CreatorId`, etc. properties that [auditing entities](/ddd/entities-and-aggregates#auditing) carry — so DTOs map 1:1 to entity audit fields:

```csharp
// CreationAuditedEntityDto.cs ─── adds CreationTime + CreatorId
[Serializable]
public abstract class CreationAuditedEntityDto : EntityDto, ICreationAuditedObject
{
    public DateTime CreationTime { get; set; }
    public Guid? CreatorId { get; set; }
}

// AuditedEntityDto.cs ─── adds LastModificationTime + LastModifierId
[Serializable]
public abstract class AuditedEntityDto : CreationAuditedEntityDto, IAuditedObject
{
    public DateTime? LastModificationTime { get; set; }
    public Guid? LastModifierId { get; set; }
}

// FullAuditedEntityDto.cs ─── adds IsDeleted/DeleterId/DeletionTime
[Serializable]
public abstract class FullAuditedEntityDto : AuditedEntityDto, IFullAuditedObject
{
    public bool IsDeleted { get; set; }
    public Guid? DeleterId { get; set; }
    public DateTime? DeletionTime { get; set; }
}
```

Each has a generic `<TPrimaryKey>` overload and a `*WithUser` variant that adds a navigation to the creator/modifier user DTO (used by Identity, Audit Logging, etc.). The extensible counterparts (`ExtensibleCreationAuditedEntityDto`, `ExtensibleAuditedEntityDto`, `ExtensibleFullAuditedEntityDto`) inherit from `ExtensibleObject` instead of `EntityDto`.

### List and paging DTOs

The list/paging DTOs let app services standardise their return types:

```csharp
// IListResult.cs
public interface IListResult<T>
{
    IReadOnlyList<T> Items { get; set; }
}

// IPagedResult.cs
public interface IPagedResult<T> : IListResult<T>, IHasTotalCount { }

// PagedResultDto.cs
public class PagedResultDto<T> : ListResultDto<T>, IPagedResult<T>
{
    public long TotalCount { get; set; }
    public PagedResultDto() { }
    public PagedResultDto(long totalCount, IReadOnlyList<T> items) : base(items) { TotalCount = totalCount; }
}
```

There is also an `ExtensiblePagedResultDto<T>` for the extensible variants.

### Input DTOs

The same folder ships the request DTOs that the `GetListAsync` overloads take:

```csharp
// LimitedResultRequestDto.cs
[Serializable]
public class LimitedResultRequestDto : ILimitedResultRequest
{
    public static int DefaultMaxResultCount { get; set; } = 10;
    public static int MaxMaxResultCount { get; set; } = 1000;

    [Range(1, int.MaxValue)]
    public virtual int MaxResultCount { get; set; } = DefaultMaxResultCount;
}

// PagedResultRequestDto.cs
[Serializable]
public class PagedResultRequestDto : LimitedResultRequestDto, IPagedResultRequest
{
    public virtual int SkipCount { get; set; }
}

// PagedAndSortedResultRequestDto.cs
[Serializable]
public class PagedAndSortedResultRequestDto : PagedResultRequestDto, IPagedAndSortedResultRequest
{
    public virtual string? Sorting { get; set; }
}
```

`MaxResultCount` is bounded by the static `DefaultMaxResultCount`/`MaxMaxResultCount` properties — clients can't request unbounded pages. The matching `ExtensibleLimitedResultRequestDto`/`ExtensiblePagedResultRequestDto`/`ExtensiblePagedAndSortedResultRequestDto` exist for extensible APIs.

### Why two parallel hierarchies?

The plain DTOs predate `ExtensibleObject`. Newer ABP modules tend to use the extensible variants exclusively because they let downstream solutions add tenant-specific custom fields, drive form generation from `ExtensionPropertyConfiguration` (in `framework/src/Volo.Abp.ObjectExtending/Volo/Abp/ObjectExtending/Modularity/`), and round-trip extra data through the HTTP API. See [Object Extending](/ddd/object-extending) for the full ObjectExtensionManager story.

## Localization resource

The `Localization/Resources/AbpDdd` folder contains the JSON files for module-level error messages (e.g. "There is no entity with the given id."). The static resource class `AbpDddApplicationContractsResource` is the strongly-typed handle:

```csharp
// framework/src/Volo.Abp.Ddd.Application.Contracts/Volo/Abp/Application/Localization/Resources/AbpDdd/AbpDddApplicationContractsResource.cs
[LocalizationResourceName("AbpDdd")]
public class AbpDddApplicationContractsResource { }
```

Application services pick this up automatically through `ApplicationService.L` (the `IStringLocalizer` property in [Application](/ddd/application)).

## Naming conventions

| Concept | Convention |
| --- | --- |
| Contract interface | `IXxxAppService` in `Application.Contracts` |
| Implementation | `XxxAppService : ApplicationService, IXxxAppService` in `Application` |
| Read DTO | `XxxDto` (extends `ExtensibleAuditedEntityDto<TKey>` if applicable) |
| Create DTO | `CreateXxxDto` |
| Update DTO | `UpdateXxxDto` |
| Combined Create+Update | `CreateUpdateXxxDto` |
| List request | `GetXxxListInput : PagedAndSortedResultRequestDto` |

When generated through ABP CLI/Studio, every module follows exactly this convention. See [Application](/ddd/application) for the concrete glue that links these DTOs to the entity.

<Tip>
A clean rule when you contribute to ABP modules: if a type is referenced from a Blazor WASM project or a generated HTTP client, it belongs in `Application.Contracts`. If it touches `IRepository`, `DbContext`, `IDomainService`, or `IObjectMapper`, it belongs in `Application`.
</Tip>
