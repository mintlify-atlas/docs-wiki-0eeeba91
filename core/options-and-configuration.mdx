ABP is built on the standard Microsoft.Extensions.Options pattern, but it adds two ergonomics that modules rely on heavily: a **`PreConfigure<T>` phase** that runs before any `Configure<T>` call, and an `AbpOptionsFactory<T>` that runs setups, post-configures, and validations in a deterministic order. The relevant source is in [`framework/src/Volo.Abp.Core/Volo/Abp/Options/`](https://github.com/abpframework/abp/tree/dev/framework/src/Volo.Abp.Core/Volo/Abp/Options) and the `Microsoft/Extensions/DependencyInjection/` extensions tree.

## Where `IConfiguration` comes from

Inside a module, `context.Configuration` is always available:

```csharp
public class MyModule : AbpModule
{
    public override void ConfigureServices(ServiceConfigurationContext context)
    {
        var sec = context.Configuration.GetSection("MyModule");
        Configure<MyModuleOptions>(sec);
    }
}
```

`ServiceConfigurationContext.Configuration` is a lazy getter that delegates to `services.GetConfiguration()`:

```csharp
// framework/src/Volo.Abp.Core/Microsoft/Extensions/DependencyInjection/ServiceCollectionConfigurationExtensions.cs
[NotNull]
public static IConfiguration GetConfiguration(this IServiceCollection services)
{
    return services.GetConfigurationOrNull() ??
           throw new AbpException("Could not find an implementation of " +
               typeof(IConfiguration).AssemblyQualifiedName + " in the service collection.");
}

[CanBeNull]
public static IConfiguration? GetConfigurationOrNull(this IServiceCollection services)
{
    var hostBuilderContext = services.GetSingletonInstanceOrNull<HostBuilderContext>();
    if (hostBuilderContext?.Configuration != null)
    {
        return hostBuilderContext.Configuration as IConfigurationRoot;
    }

    return services.GetSingletonInstanceOrNull<IConfiguration>();
}
```

Two sources are consulted:

1. **`HostBuilderContext.Configuration`** — present when ABP runs inside the .NET generic host (ASP.NET Core, Worker Services). This is `IConfigurationRoot` and reflects `appsettings.json`, environment variables, command-line args, and any in-memory sources the host added.
2. **`IConfiguration` registered as a singleton** — the standalone case (console apps, tests). You add it yourself via `services.ReplaceConfiguration(BuildConfiguration())` or by registering an `IConfiguration` singleton.

If neither exists, ABP throws on first read. The fallback is `AbpApplicationCreationOptions.Configuration` (an `AbpConfigurationBuilderOptions`), which `AddCoreAbpServices` materializes into an `IConfigurationRoot` for the standalone path.

```csharp
public static IServiceCollection ReplaceConfiguration(this IServiceCollection services, IConfiguration configuration)
{
    return services.Replace(ServiceDescriptor.Singleton<IConfiguration>(configuration));
}
```

<Note>
  The non-null `GetConfiguration` is the right call inside modules. Reach for `GetConfigurationOrNull` only when writing extension methods that should degrade gracefully (e.g. the conventional registrar pipeline).
</Note>

## The three-phase options pattern

ABP modules expose three calls — `PreConfigure<T>`, `Configure<T>`, and `PostConfigure<T>` — each mapped to a phase in `AbpApplicationBase.ConfigureServices` (see [Application lifecycle](/core/abp-application-base#configureservices--three-passes-over-every-module)).

| Phase | Verb on `AbpModule` | Backed by | Reads happen when |
| --- | --- | --- | --- |
| Pre | `PreConfigure<T>(action)` | `services.PreConfigure<T>(action)` → `PreConfigureActionList<T>` | **Immediately** when another module calls `services.ExecutePreConfiguredActions<T>()` during its own `ConfigureServices`. |
| Main | `Configure<T>(action)` / `Configure<T>(IConfiguration)` | `services.Configure<T>(...)` (the stock Microsoft call) | When `IOptions<T>` is first resolved at runtime. |
| Post | `PostConfigure<T>(action)` | `services.PostConfigure<T>(action)` | After every `Configure<T>` setup, at the moment of resolution. |

### `PreConfigure<T>` is *not* `Microsoft.Extensions.Options.IConfigureOptions<T>`

The big subtle difference: **Microsoft's** `Configure<T>` queues an `IConfigureOptions<T>` callback that fires lazily, when `IOptions<T>` is requested. ABP's `PreConfigure<T>` is **eager** — it stores delegates into a `PreConfigureActionList<T>`, and *some other module* later calls `services.ExecutePreConfiguredActions<T>()` to drain the list into a `T` instance synchronously during `ConfigureServices`.

```csharp
// framework/src/Volo.Abp.Core/Microsoft/Extensions/DependencyInjection/ServiceCollectionPreConfigureExtensions.cs
public static IServiceCollection PreConfigure<TOptions>(this IServiceCollection services, Action<TOptions> optionsAction)
{
    services.GetPreConfigureActions<TOptions>().Add(optionsAction);
    return services;
}

public static TOptions ExecutePreConfiguredActions<TOptions>(this IServiceCollection services)
    where TOptions : new()
{
    return services.ExecutePreConfiguredActions(new TOptions());
}

public static TOptions ExecutePreConfiguredActions<TOptions>(this IServiceCollection services, TOptions options)
{
    services.GetPreConfigureActions<TOptions>().Configure(options);
    return options;
}

public static PreConfigureActionList<TOptions> GetPreConfigureActions<TOptions>(this IServiceCollection services)
{
    var actionList = services.GetSingletonInstanceOrNull<IObjectAccessor<PreConfigureActionList<TOptions>>>()?.Value;
    if (actionList == null)
    {
        actionList = new PreConfigureActionList<TOptions>();
        services.AddObjectAccessor(actionList);
    }

    return actionList;
}
```

The action list is stored via [`IObjectAccessor<T>`](/core/dependency-injection#iobjectaccessort--the-late-binding-holder), so it survives across the pre/main/post passes as a single instance.

The idiomatic usage pattern is:

```csharp
public class MyAuthModule : AbpModule
{
    public override void PreConfigureServices(ServiceConfigurationContext context)
    {
        // Module A: contribute schemes
        PreConfigure<AbpClaimsServiceOptions>(o => o.RequestedClaims.Add("permissions"));
    }
}

public class MyHostModule : AbpModule
{
    public override void ConfigureServices(ServiceConfigurationContext context)
    {
        // Module B: consume the accumulated config
        var preOptions = context.Services.ExecutePreConfiguredActions<AbpClaimsServiceOptions>();
        // ... use preOptions to drive further registrations
    }
}
```

Why bother instead of plain `Configure<T>`? Because **the consumer needs the value during `ConfigureServices`**, not at runtime resolution. The middleware module wants to look at the list of claims and decide whether to add a particular middleware. `Configure<T>` would defer that decision until far too late.

### `Configure<T>` — the workhorse

The `AbpModule` helpers are thin wrappers:

```csharp
// framework/src/Volo.Abp.Core/Volo/Abp/Modularity/AbpModule.cs
protected void Configure<TOptions>(Action<TOptions> configureOptions) where TOptions : class
    => ServiceConfigurationContext.Services.Configure(configureOptions);

protected void Configure<TOptions>(string name, Action<TOptions> configureOptions) where TOptions : class
    => ServiceConfigurationContext.Services.Configure(name, configureOptions);

protected void Configure<TOptions>(IConfiguration configuration) where TOptions : class
    => ServiceConfigurationContext.Services.Configure<TOptions>(configuration);

protected void Configure<TOptions>(IConfiguration configuration, Action<BinderOptions> configureBinder) where TOptions : class
    => ServiceConfigurationContext.Services.Configure<TOptions>(configuration, configureBinder);

protected void Configure<TOptions>(string name, IConfiguration configuration) where TOptions : class
    => ServiceConfigurationContext.Services.Configure<TOptions>(name, configuration);
```

These call straight through to `Microsoft.Extensions.DependencyInjection`'s built-in `Configure` extension. Nothing magical — but consistency matters: a module that calls `services.Configure<T>(...)` directly behaves identically to one that calls `Configure<T>(...)` on the base class.

### `PostConfigure<T>` and `PostConfigureAll<T>`

```csharp
protected void PostConfigure<TOptions>(Action<TOptions> configureOptions) where TOptions : class
    => ServiceConfigurationContext.Services.PostConfigure(configureOptions);

protected void PostConfigureAll<TOptions>(Action<TOptions> configureOptions) where TOptions : class
    => ServiceConfigurationContext.Services.PostConfigureAll(configureOptions);
```

`PostConfigureAll<T>` runs against every *named* instance of `T` — useful when you're configuring a multi-tenant or multi-provider options set (e.g. an authentication scheme per tenant) and want one fix-up applied to all of them.

## `AbpOptionsFactory<TOptions>`

The framework substitutes Microsoft's stock `OptionsFactory<T>` with `AbpOptionsFactory<T>` so it can guarantee a specific ordering of setups → post-configures → validations:

```csharp
// framework/src/Volo.Abp.Core/Volo/Abp/Options/AbpOptionsFactory.cs
public class AbpOptionsFactory<TOptions> : IOptionsFactory<TOptions> where TOptions : class, new()
{
    private readonly IConfigureOptions<TOptions>[] _setups;
    private readonly IPostConfigureOptions<TOptions>[] _postConfigures;
    private readonly IValidateOptions<TOptions>[] _validations;

    public AbpOptionsFactory(
        IEnumerable<IConfigureOptions<TOptions>> setups,
        IEnumerable<IPostConfigureOptions<TOptions>> postConfigures,
        IEnumerable<IValidateOptions<TOptions>> validations)
    {
        _setups = setups as IConfigureOptions<TOptions>[] ?? new List<IConfigureOptions<TOptions>>(setups).ToArray();
        _postConfigures = postConfigures as IPostConfigureOptions<TOptions>[] ?? new List<IPostConfigureOptions<TOptions>>(postConfigures).ToArray();
        _validations = validations as IValidateOptions<TOptions>[] ?? new List<IValidateOptions<TOptions>>(validations).ToArray();
    }

    public virtual TOptions Create(string name)
    {
        var options = CreateInstance(name);

        ConfigureOptions(name, options);
        PostConfigureOptions(name, options);
        ValidateOptions(name, options);

        return options;
    }

    protected virtual void ConfigureOptions(string name, TOptions options)
    {
        foreach (var setup in _setups)
        {
            if (setup is IConfigureNamedOptions<TOptions> namedSetup)
            {
                namedSetup.Configure(name, options);
            }
            else if (name == Microsoft.Extensions.Options.Options.DefaultName)
            {
                setup.Configure(options);
            }
        }
    }

    protected virtual void PostConfigureOptions(string name, TOptions options) { /* ... */ }
    protected virtual void ValidateOptions(string name, TOptions options)       { /* ... */ }
}
```

The `Create` pass: snapshot all setups, post-configures, and validators captured at the moment of construction, and apply them in that order. Validation failures throw `OptionsValidationException`.

## `AbpDynamicOptionsManager<T>`

For options that need to be **recomputed per scope** (e.g. settings that depend on the current tenant), the framework ships an abstract base:

```csharp
// framework/src/Volo.Abp.Core/Volo/Abp/Options/AbpDynamicOptionsManager.cs
public abstract class AbpDynamicOptionsManager<T> : OptionsManager<T>
    where T : class
{
    protected AbpDynamicOptionsManager(IOptionsFactory<T> factory)
        : base(factory)
    {
    }

    public Task SetAsync() => SetAsync(Microsoft.Extensions.Options.Options.DefaultName);

    public virtual Task SetAsync(string name)
    {
        return OverrideOptionsAsync(name, base.Get(name));
    }

    protected abstract Task OverrideOptionsAsync(string name, T options);
}
```

A concrete subclass overrides `OverrideOptionsAsync` to mutate `options` based on ambient state (current tenant, current user, current culture), then a consumer calls `await myDynamicManager.SetAsync()` to apply the override before reading `Get(name)`. The `Volo.Abp.Settings` module uses this pattern to surface per-tenant settings without rebuilding the options pipeline.

## `ServiceCollectionConfigurationExtensions` quick reference

| Extension | Purpose |
| --- | --- |
| `services.ReplaceConfiguration(config)` | Swap the registered `IConfiguration` singleton — used by tests and the standalone host path. |
| `services.GetConfiguration()` | Throws if no configuration is registered. |
| `services.GetConfigurationOrNull()` | Returns null instead of throwing — used by registrars. |

The lookup order is `HostBuilderContext.Configuration` → singleton `IConfiguration`. The framework prefers the host builder context because it gives `IConfigurationRoot`, which is needed for `IConfiguration.Reload()` scenarios.

## Patterns to internalize

<AccordionGroup>
  <Accordion title="Use Configure<T> in ConfigureServices, never in OnApplicationInitialization">
    The runtime hooks (`OnApplicationInitialization` and friends) receive an `ApplicationInitializationContext` with a *scoped* `IServiceProvider`. You cannot mutate `IServiceCollection` from there — the container is already built. All option setup belongs in `ConfigureServices` / `PreConfigureServices` / `PostConfigureServices`.
  </Accordion>
  <Accordion title="Use PreConfigure<T> when another module needs to read the value during ConfigureServices">
    The classic case is an auth/authorization module that exposes a `List<Policy>`. A downstream module wants to look at that list and register middleware accordingly. Plain `Configure<T>` defers the evaluation; `PreConfigure<T>` + `ExecutePreConfiguredActions<T>()` resolves it immediately.
  </Accordion>
  <Accordion title="Bind IConfiguration sections via Configure<T>(IConfiguration)">
    `Configure<MyOptions>(context.Configuration.GetSection("MyKey"))` is the right shape. Avoid manually `Bind(...)`ing into a local `new MyOptions()`; the framework's snapshot semantics are based on the options factory.
  </Accordion>
  <Accordion title="PostConfigure<T> is for cross-module fix-up">
    If your module is the host (last in dependency order) and you need to add an item that other modules contributed via `Configure<T>`, use `PostConfigure<T>`. Otherwise your `Configure<T>` callback would run *before* theirs, since modules iterate in dependency order.
  </Accordion>
</AccordionGroup>

## `Microsoft/Extensions/DependencyInjection/` extensions catalog

Quick reference for the ABP-shipped extensions used throughout option setup:

| Extension | File | Purpose |
| --- | --- | --- |
| `services.PreConfigure<T>(action)` | `ServiceCollectionPreConfigureExtensions.cs` | Queue an eager action into `PreConfigureActionList<T>`. |
| `services.ExecutePreConfiguredActions<T>()` | same | Materialize a `T` by running all queued actions. |
| `services.AddAbpDynamicOptions<TOptions, TManager>()` | `ServiceCollectionDynamicOptionsManagerExtensions.cs` | Register a dynamic options manager and swap the default `IOptionsManager<T>` for it. |
| `services.AddObjectAccessor(instance)` / `TryAddObjectAccessor<T>()` | `ServiceCollectionObjectAccessorExtensions.cs` | Publish a singleton holder that can be filled in later. |
| `services.GetSingletonInstance<T>()` / `GetSingletonInstanceOrNull<T>()` | `ServiceCollectionCommonExtensions.cs` | Pull an instance that has already been registered as a singleton-of-instance (with `services.AddSingleton<T>(instance)`). |
