ABP ships **two** Oracle Database adapters: `Volo.Abp.EntityFrameworkCore.Oracle` (using Oracle's official ODP.NET via `Oracle.EntityFrameworkCore`) and `Volo.Abp.EntityFrameworkCore.Oracle.Devart` (using Devart's commercial `dotConnect for Oracle`). They are independent NuGet packages â€” pick one. Sources are at `framework/src/Volo.Abp.EntityFrameworkCore.Oracle/` and `framework/src/Volo.Abp.EntityFrameworkCore.Oracle.Devart/`.

Both adapters are tiny â€” module, `UseOracle` extension, modelâ€‘builder helper, connectionâ€‘string checker. The substantial behavior lives in [`Volo.Abp.EntityFrameworkCore`](/data/entityframeworkcore). This page covers both side by side.

## Why two packages?

| Adapter package                                  | EF Core provider                                | `Database.ProviderName`               | Notes                                                              |
| ------------------------------------------------ | ----------------------------------------------- | ------------------------------------- | ------------------------------------------------------------------ |
| `Volo.Abp.EntityFrameworkCore.Oracle`             | `Oracle.EntityFrameworkCore` (Oracle ODP.NET)   | `Oracle.EntityFrameworkCore`           | Vendorâ€‘official, royaltyâ€‘free, widely deployed.                     |
| `Volo.Abp.EntityFrameworkCore.Oracle.Devart`      | `Devart.Data.Oracle.Entity.EFCore` (dotConnect) | `Devart.Data.Oracle.Entity.EFCore`     | Commercial; supports advanced Devart features (data sources, etc.). |

`AbpDbContext.GetDatabaseProviderOrNull` recognizes both provider names as `EfCoreDatabaseProvider.Oracle`:

```csharp
case "Oracle.EntityFrameworkCore":
case "Devart.Data.Oracle.Entity.EFCore":
    return EfCoreDatabaseProvider.Oracle;
```

## ODP.NET adapter (`Volo.Abp.EntityFrameworkCore.Oracle`)

### Module

```csharp
// framework/src/Volo.Abp.EntityFrameworkCore.Oracle/Volo/Abp/EntityFrameworkCore/Oracle/AbpEntityFrameworkCoreOracleModule.cs
[DependsOn(typeof(AbpEntityFrameworkCoreModule))]
public class AbpEntityFrameworkCoreOracleModule : AbpModule
{
    public override void ConfigureServices(ServiceConfigurationContext context)
    {
        Configure<AbpSequentialGuidGeneratorOptions>(options =>
        {
            if (options.DefaultSequentialGuidType == null)
            {
                options.DefaultSequentialGuidType = SequentialGuidType.SequentialAsBinary;
            }
        });

        Configure<AbpEfCoreGlobalFilterOptions>(options =>
        {
            options.UseDbFunction = true;
        });
    }
}
```

`SequentialGuidType.SequentialAsBinary` is chosen because Oracle stores GUIDs as `RAW(16)` â€” raw binary, sorted by byte. Sequential bytes at the high end â†’ better clustering on the PK.

### UseOracle extension

```csharp
// framework/src/Volo.Abp.EntityFrameworkCore.Oracle/Volo/Abp/EntityFrameworkCore/AbpDbContextOptionsOracleExtensions.cs
public static class AbpDbContextOptionsOracleExtensions
{
    public static void UseOracle(
        [NotNull] this AbpDbContextOptions options,
        Action<OracleDbContextOptionsBuilder>? oracleOptionsAction = null)
    {
        options.Configure(context =>
        {
            context.UseOracle(oracleOptionsAction);
        });
    }

    public static void UseOracle<TDbContext>(
        [NotNull] this AbpDbContextOptions options,
        Action<OracleDbContextOptionsBuilder>? oracleOptionsAction = null)
        where TDbContext : AbpDbContext<TDbContext>
    {
        options.Configure<TDbContext>(context =>
        {
            context.UseOracle(oracleOptionsAction);
        });
    }
}
```

`OracleDbContextOptionsBuilder` is from `Oracle.EntityFrameworkCore.Infrastructure` â€” ODP.NET's options builder.

### Connection-string checker

```csharp
// framework/src/Volo.Abp.EntityFrameworkCore.Oracle/Volo/Abp/EntityFrameworkCore/ConnectionStrings/OracleConnectionStringChecker.cs
[Dependency(ReplaceServices = true)]
public class OracleConnectionStringChecker : IConnectionStringChecker, ITransientDependency
{
    public virtual async Task<AbpConnectionStringCheckResult> CheckAsync(string connectionString)
    {
        var result = new AbpConnectionStringCheckResult();
        try
        {
            var connString = new OracleConnectionStringBuilder(connectionString)
            {
                ConnectionTimeout = 1
            };

            await using var conn = new OracleConnection(connString.ConnectionString);
            await conn.OpenAsync();
            result.Connected = true;
            result.DatabaseExists = true;
            await conn.CloseAsync();
            return result;
        }
        catch (Exception) { return result; }
    }
}
```

<Note>
Notice this checker sets `DatabaseExists = true` whenever it can open the connection â€” Oracle doesn't have a "database" concept comparable to SQL Server's; what you're connecting to is the *service / SID*. If you can open it, the user's default schema is implicit. So the SaaS host only uses the `Connected` flag for Oracle.
</Note>

## Devart adapter (`Volo.Abp.EntityFrameworkCore.Oracle.Devart`)

### Module

```csharp
// framework/src/Volo.Abp.EntityFrameworkCore.Oracle.Devart/Volo/Abp/EntityFrameworkCore/Oracle/Devart/AbpEntityFrameworkCoreOracleDevartModule.cs
[DependsOn(typeof(AbpEntityFrameworkCoreModule))]
public class AbpEntityFrameworkCoreOracleDevartModule : AbpModule
{
    public override void ConfigureServices(ServiceConfigurationContext context)
    {
        Configure<AbpSequentialGuidGeneratorOptions>(options =>
        {
            if (options.DefaultSequentialGuidType == null)
            {
                options.DefaultSequentialGuidType = SequentialGuidType.SequentialAsBinary;
            }
        });

        Configure<AbpEfCoreGlobalFilterOptions>(options =>
        {
            options.UseDbFunction = true;
        });
    }
}
```

Same GUID strategy and DB function filter setting as ODP.NET.

### UseOracle extension (Devart)

```csharp
// framework/src/Volo.Abp.EntityFrameworkCore.Oracle.Devart/Volo/Abp/EntityFrameworkCore/AbpDbContextOptionsOracleDevartExtensions.cs
public static class AbpDbContextOptionsOracleDevartExtensions
{
    public static void UseOracle(
        [NotNull] this AbpDbContextOptions options,
        Action<OracleDbContextOptionsBuilder>? oracleOptionsAction = null,
        bool useExistingConnectionIfAvailable = false)
    {
        options.Configure(context =>
        {
            context.UseOracle(oracleOptionsAction, useExistingConnectionIfAvailable);
        });
    }

    public static void UseOracle<TDbContext>(
        [NotNull] this AbpDbContextOptions options,
        Action<OracleDbContextOptionsBuilder>? oracleOptionsAction = null,
        bool useExistingConnectionIfAvailable = false)
        where TDbContext : AbpDbContext<TDbContext>
    {
        options.Configure<TDbContext>(context =>
        {
            context.UseOracle(oracleOptionsAction, useExistingConnectionIfAvailable);
        });
    }
}
```

Three differences from the ODP.NET version:

- `OracleDbContextOptionsBuilder` is the **Devart** options builder (`Devart.Data.Oracle.Entity`).
- The signature takes a `useExistingConnectionIfAvailable` flag â€” Devart's stack supports reusing an existing ADO.NET connection.
- The underlying extension is Devart's `UseOracle(...)`.

### Connection-string checker (Devart)

```csharp
// framework/src/Volo.Abp.EntityFrameworkCore.Oracle.Devart/Volo/Abp/EntityFrameworkCore/ConnectionStrings/OracleDevartConnectionStringChecker.cs
[Dependency(ReplaceServices = true)]
public class OracleDevartConnectionStringChecker : IConnectionStringChecker, ITransientDependency
{
    public virtual async Task<AbpConnectionStringCheckResult> CheckAsync(string connectionString)
    {
        var result = new AbpConnectionStringCheckResult();
        try
        {
            var connString = new OracleConnectionStringBuilder(connectionString)
            {
                ConnectionTimeout = 1
            };

            await using var conn = new OracleConnection(connString.ConnectionString);
            await conn.OpenAsync();
            result.Connected = true;
            result.DatabaseExists = true;
            await conn.CloseAsync();
            return result;
        }
        catch (Exception) { return result; }
    }
}
```

Uses `Devart.Data.Oracle.OracleConnection` instead of ODP.NET's. Same probe shape.

<Warning>
Like the MySQL story, you **cannot** install both Oracle adapter packages in the same project. Both expose `UseOracle(this AbpDbContextOptions)` and both register an `IConnectionStringChecker` with `ReplaceServices = true`. Pick one.
</Warning>

## Module wiring

ODP.NET:

```xml
<PackageReference Include="Volo.Abp.EntityFrameworkCore.Oracle" Version="..." />
```

```csharp
[DependsOn(typeof(AbpEntityFrameworkCoreOracleModule))]
public class MyAppEntityFrameworkCoreModule : AbpModule
{
    public override void ConfigureServices(ServiceConfigurationContext context)
    {
        Configure<AbpDbContextOptions>(options =>
        {
            options.UseOracle();
        });

        context.Services.AddAbpDbContext<MyAppDbContext>(opts =>
        {
            opts.AddDefaultRepositories(includeAllEntities: true);
        });
    }
}
```

## Notes and pitfalls

<AccordionGroup>
  <Accordion title="Identifier length limit">
    Preâ€‘12.2 Oracle limits identifiers to 30 characters. ABP module tables (e.g. `AbpPermissionManagement_PermissionGroups`) can exceed that â€” verify your target Oracle version supports 128â€‘char identifiers, or override table names in `OnModelCreating`.
  </Accordion>
  <Accordion title="Boolean columns">
    Oracle has no `BOOLEAN` SQL type for columns. ABP's `IsDeleted`, `IsActive` etc. are stored as `NUMBER(1)`. EF Core's Oracle providers do the conversion automatically.
  </Accordion>
  <Accordion title="Sequences vs identity">
    Oracle 12c+ supports `IDENTITY` but ABP entities mostly use `Guid` keys generated clientâ€‘side by `IGuidGenerator`, so sequences are rarely needed for the framework's own tables.
  </Accordion>
  <Accordion title="Connect timeout vs the checker">
    The checker sets `ConnectionTimeout = 1` (1 second). On a slow VPN this can yield false negatives. Override the checker by registering your own `IConnectionStringChecker` with `[Dependency(ReplaceServices = true)]` if you need a longer probe.
  </Accordion>
</AccordionGroup>

## Choosing between the two

<AccordionGroup>
  <Accordion title="Pick ODP.NET (Volo.Abp.EntityFrameworkCore.Oracle) if...">
    - You want the vendorâ€‘official driver maintained by Oracle.
    - No license cost beyond Oracle DB itself.
    - You're integrating with Oracle Cloud or RAC features that ODP.NET supports natively.
  </Accordion>
  <Accordion title="Pick Devart (Volo.Abp.EntityFrameworkCore.Oracle.Devart) if...">
    - You already license dotConnect across your portfolio.
    - You need Devartâ€‘specific features (Direct mode without an Oracle Client install, advanced auditing, sophisticated EF Core extensions).
    - Crossâ€‘platform deployment without an Oracle Instant Client matters.
  </Accordion>
  <Accordion title="Do NOT install both">
    Both register `UseOracle(this AbpDbContextOptions)` and both `[Dependency(ReplaceServices = true)]` an `IConnectionStringChecker`. Pick one per project.
  </Accordion>
</AccordionGroup>
