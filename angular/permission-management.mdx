## The module

There are no routes ŌĆö applications never navigate to a "permission management" page. Instead the component is dropped inside another page (a users list, a roles list, a tenants list, ŌĆ”) and bound to a `[(visible)]` toggle and a `providerName` / `providerKey` pair.

The component lives in `components/permission-management.component.ts`:

### Inputs

| Input | Purpose |
| --- | --- |
| `providerName` | Permission provider id ŌĆö typically `'R'` (role), `'U'` (user), `'T'` (tenant), or `'C'` (client). |
| `providerKey` | The identifier inside the provider ŌĆö a user id, a role name, a tenant id, ŌĆ” |

| `visible` (twoŌĆæway) | Bind to a `boolean` to open and close the modal. |

### Outputs

`visibleChange` fires when the modal opens or closes ŌĆö usually used to clear the host's `providerKey` after dismissal.

### Visibility logic

Opening the modal lazily fetches the permission tree:

```ts
@Input()
get visible(): boolean { return this._visible; }
set visible(value: boolean) {
  if (value === this._visible) { return; }
  if (value) {
    this.openModal().subscribe(() => {
      this._visible = true;
      this.visibleChange.emit(true);
      concat(this.selectAllInAllTabsRef.changes, this.selectAllInThisTabsRef.changes)
        .pipe(take(1))
        .subscribe(() => { this.initModal(); });
    });
  } else {
    this.setSelectedGroup(null);
    this._visible = false;
    this.visibleChange.emit(false);
    this.filter.set('');
  }
}
```

`openModal()` calls `PermissionsService.get(providerName, providerKey)` which targets the backend endpoint described in [Authorization ŌåÆ Permissions](/security/permissions). The response contains every `PermissionGroupDto` defined by the application, the granted state per permission, and the provider info (so the modal can show "Granted from role X" badges).

`UpdatePermissionDto[]` is built from the user's toggles and sent through `PermissionsService.update(providerName, providerKey, { permissions })` on save.

## Consuming the component

Identity's `UsersComponent` is the canonical consumer. The host page declares a `providerKey` signal that points to the selected user, then renders the modal in its template:

```html
<abp-permission-management
  *abpReplaceableTemplate="{
    inputs: { providerName: 'U', providerKey, entityDisplayName },
    outputs: { visibleChange: onVisiblePermissionManagementModal.bind(this) },
    componentKey: permissionManagementKey
  }"
  [(visible)]="isPermissionManagementVisible"
  [providerName]="'U'"
  [providerKey]="providerKey"
  [entityDisplayName]="entityDisplayName">
</abp-permission-management>
```

```ts
// Inside the host component
permissionManagementKey = ePermissionManagementComponents.PermissionManagement;

openPermissionsModal(providerKey: string, entityDisplayName: string) {
  this.providerKey = providerKey;
  this.entityDisplayName = entityDisplayName;
  this.isPermissionManagementVisible = true;
}
```

The `ePermissionManagementComponents.PermissionManagement` token lets applications swap the component for a custom permission UI via `ReplaceableComponentsService.add({ key, component })`.

## Provider names used across ABP

| Provider | Used by |
| --- | --- |
| `R` (Role) | `@abp/ng.identity` ŌĆö opens the modal from the role action menu. |
| `U` (User) | `@abp/ng.identity` ŌĆö opens the modal from the user action menu. |
| `T` (Tenant) | `@abp/ng.tenant-management` ŌĆö manage permissions inherited by every user inside a tenant. |
| `C` (Client) | OAuth client management UIs (Volo's "OpenIddict" modules). |

Behind the scenes ASP.NET Core's permission management module ([`/security/permissions`](/security/permissions)) registers a `IPermissionManagementProvider` per character ŌĆö `RolePermissionManagementProvider`, `UserPermissionManagementProvider`, ŌĆ” ŌĆö that decides whether a grant is allowed to be applied.

## DTOs and proxies

The secondary entry point `@abp/ng.permission-management/proxy` exposes the strongly typed contracts used by the modal:

```ts permission-management/proxy/.../permissions.service.ts (generated)
@Injectable({ providedIn: 'root' })
export class PermissionsService {
  apiName = 'AbpPermissionManagement';
  protected restService = inject(RestService);

  get = (providerName: string, providerKey: string, config?: Partial<Rest.Config>) =>
    this.restService.request<void, GetPermissionListResultDto>(
      {
        method: 'GET',
        url: '/api/permission-management/permissions',
        params: { providerName, providerKey },
      },
      { apiName: this.apiName, ...config },
    );

  update = (
    providerName: string,
    providerKey: string,
    input: UpdatePermissionsDto,
    config?: Partial<Rest.Config>,
  ) =>
    this.restService.request<UpdatePermissionsDto, void>(
      {
        method: 'PUT',
        url: '/api/permission-management/permissions',
        params: { providerName, providerKey },
        body: input,
      },
      { apiName: this.apiName, ...config },
    );
}
```

And the DTO shapes:

```ts
interface GetPermissionListResultDto {
  entityDisplayName: string;
  groups: PermissionGroupDto[];
}

interface PermissionGroupDto {
  name: string;
  displayName: string;
  displayNameKey?: string;
  permissions: PermissionGrantInfoDto[];
}

interface PermissionGrantInfoDto {
  name: string;
  displayName: string;
  parentName?: string;
  allowedProviders: string[];
  isGranted: boolean;
  grantedProviders: ProviderInfoDto[];
}

interface UpdatePermissionDto {
  name: string;
  isGranted: boolean;
}
```

The `grantedProviders` list lets the modal show **why** a permission is granted ("granted by role admin"). When you toggle a permission off and another provider still grants it, the modal shows the badge so the user understands the resulting effective state.


Use it with `*abpReplaceableTemplate` (so a custom permission UI receives the same `providerName`, `providerKey`, `entityDisplayName` inputs and `visibleChange` output) or `ReplaceableComponentsService.add(...)`.

## Filtering and search

The modal includes a search box backed by an Angular signal (`filter = signal('')`) and a `computed()` selector that flattens permissions matching the query across all groups. The "Select all in this tab" / "Select all in all tabs" checkboxes use `viewChild` references (`selectAllInThisTabsRef`, `selectAllInAllTabsRef`) and toggle every leaf node of the permission tree.
