<Info>
This page documents the **`Volo.Abp.IdentityServer.Domain`** and **`Volo.Abp.IdentityServer.Domain.Shared`** packages. They are persistence-agnostic — actual storage is handled by the [EF Core](/modules/identityserver/efcore) or [MongoDB](/modules/identityserver/mongodb) provider modules.

Source roots:

- `modules/identityserver/src/Volo.Abp.IdentityServer.Domain/`
- `modules/identityserver/src/Volo.Abp.IdentityServer.Domain.Shared/`
</Info>

The repositories deal in **ABP aggregate roots** (entities with `Guid` ids, audit fields, change tracking). The store adapters translate those into the **IdentityServer4 model types** (`IdentityServer4.Models.Client`, `Resource`, `Grant`) that the protocol pipeline consumes. The mapping is generated by [Mapperly](https://github.com/riok/mapperly) — see `IdentityServerMapperlyMappers.cs`.

## `AbpIdentityServerDomainModule`

The module is the entry point. It depends on the Domain.Shared module, Identity domain, security, caching, validation, background-workers and the Mapperly object mapper:

```csharp
// AbpIdentityServerDomainModule.cs
[DependsOn(
    typeof(AbpIdentityServerDomainSharedModule),
    typeof(AbpMapperlyModule),
    typeof(AbpIdentityDomainModule),
    typeof(AbpSecurityModule),
    typeof(AbpCachingModule),
    typeof(AbpValidationModule),
    typeof(AbpBackgroundWorkersModule)
)]
public class AbpIdentityServerDomainModule : AbpModule
{
    public override void ConfigureServices(ServiceConfigurationContext context)
    {
        context.Services.AddMapperlyObjectMapper<AbpIdentityServerDomainModule>();

        Configure<AbpDistributedEntityEventOptions>(options =>
        {
            options.EtoMappings.Add<ApiResource, ApiResourceEto>(typeof(AbpIdentityServerDomainModule));
            options.EtoMappings.Add<Client, ClientEto>(typeof(AbpIdentityServerDomainModule));
            options.EtoMappings.Add<DeviceFlowCodes, DeviceFlowCodesEto>(typeof(AbpIdentityServerDomainModule));
            options.EtoMappings.Add<IdentityResource, IdentityResourceEto>(typeof(AbpIdentityServerDomainModule));
        });

        Configure<AbpClaimsServiceOptions>(options =>
        {
            options.RequestedClaims.AddRange(new[]{
                AbpClaimTypes.TenantId,
                AbpClaimTypes.EditionId
            });
        });

        AddIdentityServer(context.Services);
    }
}
```

A few things to call out:

- **Distributed ETO mappings.** When a `Client` (or any of the four aggregates) changes in one node, the corresponding `ClientEto` is published on the distributed event bus so peers can invalidate their caches.
- **`AbpClaimsService`.** ABP pre-populates `RequestedClaims` with `TenantId` and `EditionId` so issued tokens carry tenant context automatically — `AbpClaimsService` overrides `IdentityServer4.Services.IClaimsService` and feeds these claims into both the identity and access token pipelines.
- **`AddIdentityServer(context.Services)`** is where the magic happens — see below.

### The `AddIdentityServer` call chain

The module builds an `IIdentityServerBuilder` from scratch rather than calling the IS4 facade. That makes it possible to remove all in-memory stores and substitute ABP ones:

```csharp
private static IIdentityServerBuilder AddIdentityServer(
    IServiceCollection services,
    AbpIdentityServerBuilderOptions abpIdentityServerBuilderOptions)
{
    services.Configure<IdentityServerOptions>(options =>
    {
        options.Events.RaiseErrorEvents = true;
        options.Events.RaiseInformationEvents = true;
        options.Events.RaiseFailureEvents = true;
        options.Events.RaiseSuccessEvents = true;
    });

    var identityServerBuilder = services.AddIdentityServerBuilder()
        .AddRequiredPlatformServices()
        .AddCoreServices()
        .AddDefaultEndpoints()
        .AddPluggableServices()
        .AddValidators()
        .AddResponseGenerators()
        .AddDefaultSecretParsers()
        .AddDefaultSecretValidators();

    if (abpIdentityServerBuilderOptions.AddIdentityServerCookieAuthentication)
    {
        identityServerBuilder.AddCookieAuthentication();
    }

    identityServerBuilder.AddInMemoryPersistedGrants();
    return identityServerBuilder;
}
```

`identityServerBuilder.AddAbpIdentityServer(builderOptions)` (called immediately after, in `AddIdentityServer(IServiceCollection)`) is the ABP-side extension that swaps in the strict redirect-URI validator, the wildcard-subdomain CORS service, the claims service, and the ASP.NET Identity user store wrappers.

Then, **only if the user has not already registered a real store**, the module installs an in-memory fallback for clients and resources from `appsettings.json`:

```csharp
if (!services.IsAdded<IClientStore>())
{
    identityServerBuilder.AddInMemoryClients(
        configuration.GetSection("IdentityServer:Clients"));
}

if (!services.IsAdded<IResourceStore>())
{
    identityServerBuilder.AddInMemoryApiResources(
        configuration.GetSection("IdentityServer:ApiResources"));
    identityServerBuilder.AddInMemoryIdentityResources(
        configuration.GetSection("IdentityServer:IdentityResources"));
}
```

The EF Core and MongoDB modules both call `builder.AddAbpStores()` in `PreConfigureServices`, which registers `ClientStore`, `ResourceStore`, `PersistedGrantStore` and `DeviceFlowStore` *before* this fallback runs — so in a real solution none of the `AddInMemory*` branches execute.

### `AbpIdentityServerBuilderOptions`

`AbpIdentityServerBuilderOptions` is a pre-configurable options object (set via `services.PreConfigure<AbpIdentityServerBuilderOptions>(...)`). The two most useful knobs:

- `AddDeveloperSigningCredential` — when `true` the module calls `AddDeveloperSigningCredential()` on the builder, producing a `tempkey.jwk`. Default `true`. Set to `false` in production and provide your own X.509 signing credential.
- `AddIdentityServerCookieAuthentication` — wraps `AddCookieAuthentication()` so the IS4 cookie scheme is registered. Set to `false` if you handle cookies elsewhere (typical when running behind the Account host).

## Aggregate roots

### `Client`

`Client` is the OpenID Connect / OAuth 2.0 **client** registration. It is a `FullAuditedAggregateRoot<Guid>` so soft-delete and audit fields come for free.

```csharp
// Volo/Abp/IdentityServer/Clients/Client.cs (excerpt)
public class Client : FullAuditedAggregateRoot<Guid>
{
    public virtual string ClientId { get; set; }
    public virtual string ClientName { get; set; }
    public virtual string Description { get; set; }
    public virtual string ClientUri { get; set; }
    public virtual string LogoUri { get; set; }
    public virtual bool Enabled { get; set; } = true;
    public virtual string ProtocolType { get; set; }
    public virtual bool RequireClientSecret { get; set; }
    public virtual bool RequireConsent { get; set; }
    public virtual bool AllowRememberConsent { get; set; }
    public virtual bool RequirePkce { get; set; }
    public virtual bool AllowAccessTokensViaBrowser { get; set; }
    public virtual string FrontChannelLogoutUri { get; set; }
    public virtual string BackChannelLogoutUri { get; set; }
    public virtual bool AllowOfflineAccess { get; set; }
    public virtual int IdentityTokenLifetime { get; set; }
    public virtual int AccessTokenLifetime { get; set; }
    public virtual int AuthorizationCodeLifetime { get; set; }
    public virtual int AbsoluteRefreshTokenLifetime { get; set; }
    public virtual int SlidingRefreshTokenLifetime { get; set; }
    public virtual int RefreshTokenUsage { get; set; }
    public virtual int RefreshTokenExpiration { get; set; }
    public virtual int AccessTokenType { get; set; }
    public virtual int DeviceCodeLifetime { get; set; } = 300;

    public virtual List<ClientScope> AllowedScopes { get; set; }
    public virtual List<ClientSecret> ClientSecrets { get; set; }
    public virtual List<ClientGrantType> AllowedGrantTypes { get; set; }
    public virtual List<ClientCorsOrigin> AllowedCorsOrigins { get; set; }
    public virtual List<ClientRedirectUri> RedirectUris { get; set; }
    public virtual List<ClientPostLogoutRedirectUri> PostLogoutRedirectUris { get; set; }
    public virtual List<ClientIdPRestriction> IdentityProviderRestrictions { get; set; }
    public virtual List<ClientClaim> Claims { get; set; }
    public virtual List<ClientProperty> Properties { get; set; }
}
```

The constructor sets sensible defaults:

```csharp
public Client(Guid id, [NotNull] string clientId) : base(id)
{
    Check.NotNull(clientId, nameof(clientId));
    ClientId = clientId;

    ProtocolType = IdentityServerConstants.ProtocolTypes.OpenIdConnect;
    RequireClientSecret = true;
    RequireConsent = false;
    AllowRememberConsent = true;
    RequirePkce = true;
    FrontChannelLogoutSessionRequired = true;
    BackChannelLogoutSessionRequired = true;
    IdentityTokenLifetime = 300;
    AccessTokenLifetime = 3600;
    AuthorizationCodeLifetime = 300;
    AbsoluteRefreshTokenLifetime = 2592000;
    SlidingRefreshTokenLifetime = 1296000;
    RefreshTokenUsage = (int)TokenUsage.OneTimeOnly;
    RefreshTokenExpiration = (int)TokenExpiration.Absolute;
    AccessTokenType = (int)IdentityServer4.Models.AccessTokenType.Jwt;
    EnableLocalLogin = true;
    ClientClaimsPrefix = "client_";
    // ... empty collection initialisers
}
```

Helper methods enforce invariants on the child collections:

```csharp
public virtual void AddGrantType([NotNull] string grantType)
    => AllowedGrantTypes.Add(new ClientGrantType(Id, grantType));

public virtual void AddSecret(
    [NotNull] string value,
    DateTime? expiration = null,
    string type = IdentityServerConstants.SecretTypes.SharedSecret,
    string description = null)
    => ClientSecrets.Add(new ClientSecret(Id, value, expiration, type, description));

public virtual ClientGrantType FindGrantType(string grantType)
    => AllowedGrantTypes.FirstOrDefault(r => r.GrantType == grantType);
```

The child entities live in the same folder and follow the IS4 schema 1:1: `ClientScope`, `ClientSecret`, `ClientGrantType`, `ClientRedirectUri`, `ClientPostLogoutRedirectUri`, `ClientCorsOrigin`, `ClientIdPRestriction`, `ClientClaim`, `ClientProperty`.

### `ApiResource`

```csharp
// Volo/Abp/IdentityServer/ApiResources/ApiResource.cs (excerpt)
public class ApiResource : FullAuditedAggregateRoot<Guid>
{
    [NotNull]
    public virtual string Name { get; protected set; }
    public virtual string DisplayName { get; set; }
    public virtual string Description { get; set; }
    public virtual bool Enabled { get; set; }
    public virtual string AllowedAccessTokenSigningAlgorithms { get; set; }
    public virtual bool ShowInDiscoveryDocument { get; set; } = true;

    public virtual List<ApiResourceSecret> Secrets { get; protected set; }
    public virtual List<ApiResourceScope> Scopes { get; protected set; }
    public virtual List<ApiResourceClaim> UserClaims { get; protected set; }
    public virtual List<ApiResourceProperty> Properties { get; protected set; }

    public ApiResource(Guid id, [NotNull] string name,
        string displayName = null, string description = null) : base(id)
    {
        Check.NotNull(name, nameof(name));
        Name = name;
        DisplayName = displayName;
        Description = description;
        Enabled = true;
        Secrets = new List<ApiResourceSecret>();
        Scopes = new List<ApiResourceScope>();
        UserClaims = new List<ApiResourceClaim>();
        Properties = new List<ApiResourceProperty>();
        Scopes.Add(new ApiResourceScope(id, name));
    }
}
```

Note that the constructor **adds a scope with the same name as the resource** — that is the IS4 convention for the "primary" scope of an API resource.

### `IdentityResource`

Mirrors `ApiResource` but for identity claims (`openid`, `profile`, `email`, `role`, ...):

```csharp
public class IdentityResource : FullAuditedAggregateRoot<Guid>
{
    public virtual string Name { get; set; }
    public virtual string DisplayName { get; set; }
    public virtual string Description { get; set; }
    public virtual bool Enabled { get; set; }
    public virtual bool Required { get; set; }
    public virtual bool Emphasize { get; set; }
    public virtual bool ShowInDiscoveryDocument { get; set; }

    public virtual List<IdentityResourceClaim> UserClaims { get; protected set; }
    public virtual List<IdentityResourceProperty> Properties { get; protected set; }
}
```

The companion `IdentityResourceDataSeeder` knows how to seed the standard set:

```csharp
public virtual async Task CreateStandardResourcesAsync()
{
    var resources = new[]
    {
        new IdentityServer4.Models.IdentityResources.OpenId(),
        new IdentityServer4.Models.IdentityResources.Profile(),
        new IdentityServer4.Models.IdentityResources.Email(),
        new IdentityServer4.Models.IdentityResources.Address(),
        new IdentityServer4.Models.IdentityResources.Phone(),
        new IdentityServer4.Models.IdentityResource("role", "Roles of the user", new[] { "role" })
    };

    foreach (var resource in resources)
    {
        foreach (var claimType in resource.UserClaims)
        {
            await AddClaimTypeIfNotExistsAsync(claimType);
        }
        await AddIdentityResourceIfNotExistsAsync(resource);
    }
}
```

The seeder also creates entries in the [Identity module's](/modules/identity/overview) `IIdentityClaimTypeRepository` so the admin UI can render those claim types.

### `PersistedGrant`

The operational store entity. It is a plain `AggregateRoot<Guid>` (no audit columns — these rows churn):

```csharp
// Volo/Abp/IdentityServer/Grants/PersistedGrant.cs
public class PersistedGrant : AggregateRoot<Guid>
{
    public virtual string Key { get; set; }
    public virtual string Type { get; set; }
    public virtual string SubjectId { get; set; }
    public virtual string SessionId { get; set; }
    public virtual string ClientId { get; set; }
    public virtual string Description { get; set; }
    public virtual DateTime CreationTime { get; set; }
    public virtual DateTime? Expiration { get; set; }
    public virtual DateTime? ConsumedTime { get; set; }
    public virtual string Data { get; set; }

    protected PersistedGrant() { }
    public PersistedGrant(Guid id) : base(id) { }
}
```

`Data` is an opaque serialised blob that IS4 controls — do not parse it. `Type` distinguishes authorization codes from refresh tokens, reference tokens and user-consent records.

### `DeviceFlowCodes`

Used for the OAuth 2.0 **device authorization grant** (`urn:ietf:params:oauth:grant-type:device_code`). It stores the user code and device code along with the captured user subject after they complete the device-flow login.

## Repository contracts

All repositories extend `IBasicRepository<TEntity, Guid>`. The methods reflect the queries that the IS4 store adapters need.

```csharp
// Volo/Abp/IdentityServer/Clients/IClientRepository.cs
public interface IClientRepository : IBasicRepository<Client, Guid>
{
    Task<Client> FindByClientIdAsync(
        [NotNull] string clientId,
        bool includeDetails = true,
        CancellationToken cancellationToken = default);

    Task<List<Client>> GetListAsync(
        string sorting, int skipCount, int maxResultCount,
        string filter = null, bool includeDetails = false,
        CancellationToken cancellationToken = default);

    Task<long> GetCountAsync(
        string filter = null,
        CancellationToken cancellationToken = default);

    Task<List<string>> GetAllDistinctAllowedCorsOriginsAsync(
        CancellationToken cancellationToken = default);

    Task<bool> CheckClientIdExistAsync(
        string clientId, Guid? expectedId = null,
        CancellationToken cancellationToken = default);
}
```

`GetAllDistinctAllowedCorsOriginsAsync` powers the `AbpWildcardSubdomainCorsPolicyService` so the dynamic CORS lookup is a single query.

```csharp
// Volo/Abp/IdentityServer/Grants/IPersistentGrantRepository.cs
public interface IPersistentGrantRepository : IBasicRepository<PersistedGrant, Guid>
{
    Task<List<PersistedGrant>> GetListAsync(
        string subjectId, string sessionId, string clientId, string type,
        bool includeDetails = false, CancellationToken cancellationToken = default);

    Task<PersistedGrant> FindByKeyAsync(
        string key, CancellationToken cancellationToken = default);

    Task<List<PersistedGrant>> GetListBySubjectIdAsync(
        string key, CancellationToken cancellationToken = default);

    Task<List<PersistedGrant>> GetListByExpirationAsync(
        DateTime maxExpirationDate, int maxResultCount,
        CancellationToken cancellationToken = default);

    Task DeleteExpirationAsync(
        DateTime maxExpirationDate,
        CancellationToken cancellationToken = default);

    Task DeleteAsync(
        string subjectId = null, string sessionId = null,
        string clientId = null, string type = null,
        CancellationToken cancellationToken = default);
}
```

`DeleteExpirationAsync` is what the background cleanup worker calls. `DeleteAsync(subjectId: ...)` is what fires when a user signs out — IS4 wipes their grants.

The other repository interfaces — `IApiResourceRepository`, `IApiScopeRepository`, `IIdentityResourceRepository`, `IDeviceFlowCodesRepository` — follow the same shape, with the queries each store adapter actually needs.

## IdentityServer4 store adapters

The store classes in `Volo.Abp.IdentityServer/.../*Store.cs` wrap a repository and implement the matching IS4 interface. For example, `ClientStore` implements `IdentityServer4.Stores.IClientStore` and maps the ABP `Client` aggregate to `IdentityServer4.Models.Client` via Mapperly (`IdentityServerMapperlyMappers`). `ResourceStore` does the same for the three resource types in a single call. `PersistedGrantStore` and `DeviceFlowStore` wrap `IPersistentGrantRepository` and `IDeviceFlowCodesRepository`.

Two ABP-specific overrides also live in the Domain layer:

- **`AbpClaimsService`** (replaces `IClaimsService`) — honours `AbpClaimsServiceOptions.RequestedClaims` so tokens carry the right tenant/edition claims.
- **`AbpStrictRedirectUriValidator`** / **`AbpWildcardSubdomainCorsPolicyService`** — enforce strict redirect-URI matching while still allowing wildcard subdomain rules for multi-tenant scenarios.

## Token cleanup

```csharp
// Volo/Abp/IdentityServer/Tokens/TokenCleanupService.cs (excerpt)
public class TokenCleanupService : ITransientDependency
{
    public TokenCleanupService(
        IPersistentGrantRepository persistentGrantRepository,
        IDeviceFlowCodesRepository deviceFlowCodesRepository,
        IOptions<TokenCleanupOptions> options)
    { ... }

    [UnitOfWork]
    public virtual async Task CleanAsync()
    {
        await RemoveGrantsAsync();
        await RemoveDeviceCodesAsync();
    }

    protected virtual Task RemoveGrantsAsync()
        => PersistentGrantRepository.DeleteExpirationAsync(DateTime.UtcNow);

    protected virtual Task RemoveDeviceCodesAsync()
        => DeviceFlowCodesRepository.DeleteExpirationAsync(DateTime.UtcNow);
}
```

The worker is added in `OnApplicationInitializationAsync`:

```csharp
public async override Task OnApplicationInitializationAsync(
    ApplicationInitializationContext context)
{
    var options = context.ServiceProvider
        .GetRequiredService<IOptions<TokenCleanupOptions>>().Value;
    if (options.IsCleanupEnabled)
    {
        await context.ServiceProvider
            .GetRequiredService<IBackgroundWorkerManager>()
            .AddAsync(
                context.ServiceProvider
                    .GetRequiredService<TokenCleanupBackgroundWorker>()
            );
    }
}
```

`TokenCleanupOptions.CleanupPeriod` defaults to `3_600_000` ms (one hour). `IsCleanupEnabled` defaults to `true`.

## Entity extension hook

`PostConfigureServices` re-applies any extra properties registered via the ABP **object-extension** system to the three configuration aggregates:

```csharp
public override void PostConfigureServices(ServiceConfigurationContext context)
{
    OneTimeRunner.Run(() =>
    {
        ModuleExtensionConfigurationHelper.ApplyEntityConfigurationToEntity(
            IdentityServerModuleExtensionConsts.ModuleName,
            IdentityServerModuleExtensionConsts.EntityNames.Client,
            typeof(Client));

        ModuleExtensionConfigurationHelper.ApplyEntityConfigurationToEntity(
            IdentityServerModuleExtensionConsts.ModuleName,
            IdentityServerModuleExtensionConsts.EntityNames.IdentityResource,
            typeof(IdentityResource));

        ModuleExtensionConfigurationHelper.ApplyEntityConfigurationToEntity(
            IdentityServerModuleExtensionConsts.ModuleName,
            IdentityServerModuleExtensionConsts.EntityNames.ApiResource,
            typeof(ApiResource));
    });
}
```

That means you can add custom columns to `Client`, `IdentityResource` and `ApiResource` from a downstream module without forking this one — the EF Core provider picks the extra properties up via `ApplyObjectExtensionMappings()` in its model-builder extension.

## Domain.Shared contents

`Volo.Abp.IdentityServer.Domain.Shared` keeps the **stable** types shared with non-server tiers (UI, application services, contracts):

- **Consts** classes like `ClientConsts` defining `MaxLength` values that drive EF Core column lengths:

  ```csharp
  // Volo/Abp/IdentityServer/Clients/ClientConsts.cs
  public class ClientConsts
  {
      public static int ClientIdMaxLength { get; set; } = 200;
      public static int ProtocolTypeMaxLength { get; set; } = 200;
      public static int ClientNameMaxLength { get; set; } = 200;
      public static int ClientUriMaxLength { get; set; } = 2000;
      public static int LogoUriMaxLength { get; set; } = 2000;
      public static int DescriptionMaxLength { get; set; } = 1000;
      // ...
  }
  ```
- **ETOs** (`ClientEto`, `ApiResourceEto`, `ApiScopeEto`, `IdentityResourceEto`, `DeviceFlowCodesEto`) used by the distributed event mappings.
- **`AbpIdentityServerResource`** localization resource (with `en` JSON resources embedded as a virtual file).
- **`UserClaimConsts`** and **`IdentityServerSecurityLogActionConsts`** for tagging audit-log entries.

The Domain.Shared module itself just wires localisation and exception code mapping:

```csharp
[DependsOn(typeof(AbpValidationModule))]
public class AbpIdentityServerDomainSharedModule : AbpModule
{
    public override void ConfigureServices(ServiceConfigurationContext context)
    {
        Configure<AbpVirtualFileSystemOptions>(options =>
        {
            options.FileSets.AddEmbedded<AbpIdentityServerDomainSharedModule>();
        });

        Configure<AbpLocalizationOptions>(options =>
        {
            options.Resources.Add<AbpIdentityServerResource>("en")
                .AddBaseTypes(typeof(AbpValidationResource))
                .AddVirtualJson("/Volo/Abp/IdentityServer/Localization/Resources");
        });

        Configure<AbpExceptionLocalizationOptions>(options =>
        {
            options.MapCodeNamespace("Volo.IdentityServer",
                typeof(AbpIdentityServerResource));
        });
    }
}
```

## Database properties

Both the EF Core and MongoDB providers consume the same constants:

```csharp
public static class AbpIdentityServerDbProperties
{
    public static string DbTablePrefix { get; set; } = "IdentityServer";
    public static string DbSchema { get; set; } = AbpCommonDbProperties.DbSchema;
    public const string ConnectionStringName = "AbpIdentityServer";
}
```

The `[ConnectionStringName("AbpIdentityServer")]` attribute on `IIdentityServerDbContext` / `AbpIdentityServerMongoDbContext` makes ABP look up the connection string under that name first, falling back to `Default`. That lets you isolate IdentityServer data on its own database.
