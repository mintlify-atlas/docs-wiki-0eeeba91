The HTTP API layer is the thinnest project in the module — one controller (`FeaturesController`) that turns `IFeatureAppService` into REST. The companion `HttpApi.Client` project wires the same interface as a static HTTP proxy so remote callers can inject `IFeatureAppService` and have ABP route the calls over the wire. This page is the route reference, the JSON contract reference, and the proxy‑wiring guide.

If you have not read the [application layer page](/modules/feature-management/application), do that first — the controller is genuinely a pass‑through, and the interesting authorisation logic lives behind the application service.

The `ClientProxies/` folder contains the generated CSharp proxies (`FeatureClientProxy`) used when `AbpFeatureManagementHttpApiClientModule` is registered. They are part of the public NuGet package; the source is generated by `abp generate-proxy` and re‑checked‑in.

## `AbpFeatureManagementHttpApiModule`

```csharp
// AbpFeatureManagementHttpApiModule.cs (essence)
[DependsOn(
    typeof(AbpFeatureManagementApplicationContractsModule),
    typeof(AbpAspNetCoreMvcModule))]
public class AbpFeatureManagementHttpApiModule : AbpModule
{
    public override void PreConfigureServices(ServiceConfigurationContext context)
    {
        PreConfigure<IMvcBuilder>(mvcBuilder =>
        {
            mvcBuilder.AddApplicationPartIfNotExists(typeof(AbpFeatureManagementHttpApiModule).Assembly);
        });
    }

    public override void ConfigureServices(ServiceConfigurationContext context)
    {
        Configure<AbpLocalizationOptions>(options =>
        {
            options.Resources
                .Get<AbpFeatureManagementResource>()
                .AddBaseTypes(typeof(AbpUiResource));
        });

        var valueValidatorFactoryOptions =
            context.Services.ExecutePreConfiguredActions<ValueValidatorFactoryOptions>();

        Configure<JsonOptions>(options =>
        {
            options.JsonSerializerOptions.Converters.AddIfNotContains(
                new StringValueTypeJsonConverter(valueValidatorFactoryOptions));
        });
    }
}
```

Three things happen at module load:

1. **`AddApplicationPartIfNotExists`** registers the assembly as an MVC application part so `FeaturesController` is discovered without you having to add it manually.
2. **Localization** layers `AbpUiResource` under `AbpFeatureManagementResource` so generic validation messages ("Required", "Invalid") resolve in the user's language.
3. **`StringValueTypeJsonConverter`** is registered. This is the converter that lets `FeatureDto.ValueType` (an `IStringValueType` polymorphic instance) round‑trip JSON — without it, deserialisation would lose the concrete type (`ToggleStringValueType` vs `FreeTextStringValueType` vs `SelectionStringValueType`).

## `FeaturesController`

The controller is a six‑line passthrough plus three actions:

```csharp
// FeaturesController.cs
[RemoteService(Name = FeatureManagementRemoteServiceConsts.RemoteServiceName)]
[Area(FeatureManagementRemoteServiceConsts.ModuleName)]
[Route("api/feature-management/features")]
public class FeaturesController : AbpControllerBase, IFeatureAppService
{
    protected IFeatureAppService FeatureAppService { get; }

    public FeaturesController(IFeatureAppService featureAppService)
    {
        FeatureAppService = featureAppService;
    }

    [HttpGet]
    public virtual Task<GetFeatureListResultDto> GetAsync(string providerName, string providerKey)
    {
        return FeatureAppService.GetAsync(providerName, providerKey);
    }

    [HttpPut]
    public virtual Task UpdateAsync(string providerName, string providerKey, UpdateFeaturesDto input)
    {
        return FeatureAppService.UpdateAsync(providerName, providerKey, input);
    }

    [HttpDelete]
    public virtual Task DeleteAsync(string providerName, string providerKey)
    {
        return FeatureAppService.DeleteAsync(providerName, providerKey);
    }
}
```

Things to note before the route reference:

- **`[RemoteService(Name = "AbpFeatureManagement")]`** — the name used by `AbpFeatureManagementHttpApiClientModule` to wire static proxies and by the dynamic JS proxy to namespace the script.
- **`[Area("featureManagement")]`** — the `ModuleName` constant, used by ABP UI conventions to group the controller under the *Feature Management* area in the auto‑generated Swagger.
- **`[Route("api/feature-management/features")]`** — the base path; every action below extends it.
- **`AbpControllerBase`** — the standard ABP base that wires localization, current user, and unit‑of‑work attributes. The `[Authorize]` from the application service propagates through the interface.

## Route reference

All three endpoints share the same base path and use `providerName` + `providerKey` as the addressing tuple:

| Method | Path | Action | Auth |
| --- | --- | --- | --- |
| `GET` | `/api/feature-management/features?providerName={pn}&providerKey={pk}` | `GetAsync` | Provider policy (see [application](/modules/feature-management/application#provider-policies)) |
| `PUT` | `/api/feature-management/features?providerName={pn}&providerKey={pk}` | `UpdateAsync` (body: `UpdateFeaturesDto`) | Provider policy |
| `DELETE` | `/api/feature-management/features?providerName={pn}&providerKey={pk}` | `DeleteAsync` | Provider policy |

`providerName` is mandatory — the `[NotNull]` annotation on `IFeatureAppService` propagates to a runtime `ArgumentException` on `null`. `providerKey` is nullable; the host record sends `providerKey=`. Conventional `Guid` keys are URL‑encoded as their canonical 36‑character form (no braces).

## Examples

### `GET` host features

```bash
curl -X GET \
  "https://your-app/api/feature-management/features?providerName=T" \
  -H "Authorization: Bearer eyJhbGc..."
```

Note: no `providerKey` query parameter ⇒ host record. Requires the `FeatureManagement.ManageHostFeatures` permission.

### `GET` tenant features

```bash
curl -X GET \
  "https://your-app/api/feature-management/features?providerName=T&providerKey=3fa85f64-5717-4562-b3fc-2c963f66afa6" \
  -H "Authorization: Bearer eyJhbGc..."
```

Requires `AbpTenantManagement.Tenants.ManageFeatures` because `providerKey` is non‑null.

### Sample response

```json
{
  "groups": [
    {
      "name": "MyApp.Reporting",
      "displayName": "Reporting",
      "features": [
        {
          "name": "MyApp.Reporting",
          "displayName": "Enable reporting",
          "value": "true",
          "provider": { "name": "E", "key": "8aa3..." },
          "description": "Allows access to the reporting module.",
          "valueType": {
            "name": "ToggleStringValueType",
            "validator": { "name": "BOOLEAN" }
          },
          "depth": 0,
          "parentName": null
        },
        {
          "name": "MyApp.Reporting.MaxRows",
          "displayName": "Max rows",
          "value": "10000",
          "provider": { "name": "D", "key": null },
          "description": null,
          "valueType": {
            "name": "FreeTextStringValueType",
            "validator": { "name": "NUMERIC", "properties": { "MinValue": "0" } }
          },
          "depth": 1,
          "parentName": "MyApp.Reporting"
        }
      ]
    }
  ]
}
```

What to spot in this payload:

- `provider.name = "E"` means the value came from the *edition*, not from a tenant override. The UI greys the input.
- `provider.name = "D"` means the value came from the `DefaultValueFeatureManagementProvider` — i.e. no override exists.
- `valueType.name` is the discriminator written by `StringValueTypeJsonConverter`. `ToggleStringValueType`, `FreeTextStringValueType`, and `SelectionStringValueType` are the built‑ins.
- `depth: 1` and `parentName: "MyApp.Reporting"` together signal "indent this row under its parent".

### `PUT` — batched update

```bash
curl -X PUT \
  "https://your-app/api/feature-management/features?providerName=T&providerKey=3fa85f64-5717-4562-b3fc-2c963f66afa6" \
  -H "Authorization: Bearer eyJhbGc..." \
  -H "Content-Type: application/json" \
  -d '{
    "features": [
      { "name": "MyApp.Reporting", "value": "true" },
      { "name": "MyApp.Reporting.MaxRows", "value": "50000" }
    ]
  }'
```

Responds with `204 No Content` on success. On a validation failure (e.g. `MaxRows` outside the numeric validator's range) the response is `400 Bad Request` with the standard ABP error envelope and `code = "AbpFeatureManagement:000001"`.

### `DELETE` — reset to inherited

```bash
curl -X DELETE \
  "https://your-app/api/feature-management/features?providerName=T&providerKey=3fa85f64-5717-4562-b3fc-2c963f66afa6" \
  -H "Authorization: Bearer eyJhbGc..."
```

Equivalent to the modal's *Reset to default* button — drops every override row for the (provider, key) pair. `204 No Content` on success.

## Error envelope

Standard `[BusinessException]` envelope:

```json
{
  "error": {
    "code": "AbpFeatureManagement:000001",
    "message": "Invalid value for the feature: Max rows!",
    "details": null,
    "data": { "DisplayName": "Max rows" }
  }
}
```

The two codes you will see in practice:

| Code | Source | Meaning |
| --- | --- | --- |
| `AbpFeatureManagement:000001` | `FeatureValueInvalidException` | Value failed `IStringValueType.Validator.IsValid`. |
| (none — `AbpAuthorizationException`) | `CheckProviderPolicy` | The current user lacks the policy mapped to the requested provider. |

## `AbpFeatureManagementHttpApiClientModule`

The client module installs **static** proxies — typed C# classes that implement `IFeatureAppService` over `HttpClient`:

```csharp
// AbpFeatureManagementHttpApiClientModule.cs
[DependsOn(
    typeof(AbpFeatureManagementApplicationContractsModule),
    typeof(AbpHttpClientModule))]
public class AbpFeatureManagementHttpApiClientModule : AbpModule
{
    public override void ConfigureServices(ServiceConfigurationContext context)
    {
        context.Services.AddStaticHttpClientProxies(
            typeof(AbpFeatureManagementApplicationContractsModule).Assembly,
            FeatureManagementRemoteServiceConsts.RemoteServiceName
        );

        Configure<AbpVirtualFileSystemOptions>(options =>
        {
            options.FileSets.AddEmbedded<AbpFeatureManagementHttpApiClientModule>();
        });
    }
}
```

`AddStaticHttpClientProxies` scans the contracts assembly for `IApplicationService`s and wires a `FeatureClientProxy` (from `ClientProxies/`) for each. After the module is added, `IFeatureAppService` resolves in DI to a proxy that issues HTTP calls — your code stays identical.

### Configuring the remote URL

Add a `RemoteServices` block to `appsettings.json`:

```json
{
  "RemoteServices": {
    "Default": {
      "BaseUrl": "https://admin.your-app/"
    },
    "AbpFeatureManagement": {
      "BaseUrl": "https://admin.your-app/"
    }
  }
}
```

The key (`AbpFeatureManagement`) matches `FeatureManagementRemoteServiceConsts.RemoteServiceName`. The fallback is the `Default` entry — if you have one admin gateway hosting every ABP API, configure only `Default` and skip the per‑service entries.

### Sample static proxy (`FeatureClientProxy`)

The checked‑in proxy mirrors the controller one‑for‑one:

```csharp
// HttpApi.Client/ClientProxies/FeatureClientProxy.Generated.cs (essence)
public partial class FeatureClientProxy : ClientProxyBase<IFeatureAppService>, IFeatureAppService
{
    public virtual async Task<GetFeatureListResultDto> GetAsync(string providerName, string providerKey)
    {
        return await RequestAsync<GetFeatureListResultDto>(
            nameof(GetAsync),
            new ClientProxyRequestTypeValue
            {
                { typeof(string), providerName },
                { typeof(string), providerKey }
            });
    }

    public virtual async Task UpdateAsync(
        string providerName, string providerKey, UpdateFeaturesDto input)
    {
        await RequestAsync(
            nameof(UpdateAsync),
            new ClientProxyRequestTypeValue
            {
                { typeof(string), providerName },
                { typeof(string), providerKey },
                { typeof(UpdateFeaturesDto), input }
            });
    }

    public virtual async Task DeleteAsync(string providerName, string providerKey)
    {
        await RequestAsync(
            nameof(DeleteAsync),
            new ClientProxyRequestTypeValue
            {
                { typeof(string), providerName },
                { typeof(string), providerKey }
            });
    }
}
```

The generator places one such file under `ClientProxies/Volo/Abp/FeatureManagement/` per application service. Re‑generate with `abp generate-proxy -t csharp` whenever you add new endpoints to the contract.

### Dynamic JS proxy

`AbpFeatureManagementWebModule` *disables* the dynamic JS proxy for this module:

```csharp
Configure<DynamicJavaScriptProxyOptions>(options =>
{
    options.DisableModule(FeatureManagementRemoteServiceConsts.ModuleName);
});
```

The MVC modal ships its own hand‑written `feature-management-modal.js` (see [Web & Blazor](/modules/feature-management/web-and-blazor)) so the dynamic proxy would be dead weight. If you build a custom JS UI, call `/api/feature-management/features` directly via `abp.ajax` or `fetch`.

## Calling the API from a typed client

A console worker that wants to read tenant features:

```csharp
public class FeatureSyncWorker
{
    private readonly IFeatureAppService _featureAppService;

    public FeatureSyncWorker(IFeatureAppService featureAppService)
    {
        _featureAppService = featureAppService;
    }

    public async Task<bool> IsReportingEnabledForTenantAsync(Guid tenantId)
    {
        var result = await _featureAppService.GetAsync(
            providerName: "T",
            providerKey: tenantId.ToString());

        var reporting = result.Groups
            .SelectMany(g => g.Features)
            .FirstOrDefault(f => f.Name == "MyApp.Reporting");

        return reporting?.Value == "true";
    }
}
```

With `AbpFeatureManagementHttpApiClientModule` registered, `_featureAppService` is the static proxy; without it (in the API host itself), the same code resolves to the in‑process `FeatureAppService`. **Write your application code against the interface; let the module wiring decide whether the call is local or remote.**
