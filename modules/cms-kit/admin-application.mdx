The admin application layer is where back-office UI talks to the domain. Each capability has at least one `*AdminAppService`, decorated with `[Authorize(CmsKitAdminPermissions.X)]` and feature attributes. DTOs and interfaces live in `Volo.CmsKit.Admin.Application.Contracts`; implementations live in `Volo.CmsKit.Admin.Application`.

- Contracts: [`modules/cms-kit/src/Volo.CmsKit.Admin.Application.Contracts/Volo/CmsKit/Admin/`](https://github.com/abpframework/abp/tree/dev/modules/cms-kit/src/Volo.CmsKit.Admin.Application.Contracts/Volo/CmsKit/Admin)

## Remote service identity

```csharp
// modules/cms-kit/src/Volo.CmsKit.Admin.Application.Contracts/Volo/CmsKit/Admin/CmsKitAdminRemoteServiceConsts.cs
public class CmsKitAdminRemoteServiceConsts
{
    public const string RemoteServiceName = "CmsKitAdmin";
    public const string ModuleName = "cms-kit-admin";
}
```

Every admin controller carries `[RemoteService(Name = "CmsKitAdmin")]` and `[Area("cms-kit-admin")]`. This is how ABP's dynamic-C#-client generates `IBlogPostAdminAppService` proxies that hit the right base address — the consuming app configures one base URL for the `CmsKitAdmin` remote service in `appsettings.json`.

## `IBlogPostAdminAppService`

```csharp
// modules/cms-kit/src/Volo.CmsKit.Admin.Application.Contracts/Volo/CmsKit/Admin/Blogs/IBlogPostAdminAppService.cs
public interface IBlogPostAdminAppService : ICrudAppService<
    BlogPostDto,
    BlogPostListDto,
    Guid,
    BlogPostGetListInput,
    CreateBlogPostDto,
    UpdateBlogPostDto>
{
    Task PublishAsync(Guid id);
    Task DraftAsync(Guid id);
    Task<BlogPostDto> CreateAndPublishAsync(CreateBlogPostDto input);
    Task SendToReviewAsync(Guid id);
    Task<BlogPostDto> CreateAndSendToReviewAsync(CreateBlogPostDto input);
    Task<bool> HasBlogPostWaitingForReviewAsync();
}
```

Two DTO types: `BlogPostDto` (full detail, returned by `GetAsync`/`CreateAsync`/`UpdateAsync`) and `BlogPostListDto` (slim, returned by `GetListAsync` — no `Content`, just metadata for the grid).

Lifecycle methods correspond to the blog post status transitions described in [Blogs](/modules/cms-kit/blogs#status-lifecycle). `CreateAndPublishAsync` and `CreateAndSendToReviewAsync` are convenience methods that do create + status change in a single request.

`HasBlogPostWaitingForReviewAsync()` powers the red badge in the admin nav.

## `BlogPostAdminAppService` — implementation excerpt

```csharp
// modules/cms-kit/src/Volo.CmsKit.Admin.Application/Volo/CmsKit/Admin/Blogs/BlogPostAdminAppService.cs
[RequiresFeature(CmsKitFeatures.BlogEnable)]
[RequiresGlobalFeature(typeof(BlogsFeature))]
[Authorize(CmsKitAdminPermissions.BlogPosts.Default)]
public class BlogPostAdminAppService : CmsKitAppServiceBase, IBlogPostAdminAppService
{
    protected BlogPostManager BlogPostManager { get; }
    protected IBlogPostRepository BlogPostRepository { get; }
    protected IBlogRepository BlogRepository { get; }
    protected ICommentRepository CommentRepository { get; }
    protected ICmsUserLookupService UserLookupService { get; }
    protected IMediaDescriptorAdminAppService MediaDescriptorAdminAppService { get; }

    public BlogPostAdminAppService(
        BlogPostManager blogPostManager,
        IBlogPostRepository blogPostRepository,
        IBlogRepository blogRepository,
        ICmsUserLookupService userLookupService,
        IMediaDescriptorAdminAppService mediaDescriptorAdminAppService,
        ICommentRepository commentRepository)
    {
        BlogPostManager = blogPostManager;
        BlogPostRepository = blogPostRepository;
        BlogRepository = blogRepository;
        UserLookupService = userLookupService;
        MediaDescriptorAdminAppService = mediaDescriptorAdminAppService;
        CommentRepository = commentRepository;
    }

    [Authorize(CmsKitAdminPermissions.BlogPosts.Create)]
    public virtual async Task<BlogPostDto> CreateAsync(CreateBlogPostDto input)
    {
        var author = await UserLookupService.GetByIdAsync(CurrentUser.GetId());
        var blog = await BlogRepository.GetAsync(input.BlogId);

        var blogPost = await BlogPostManager.CreateAsync(
            author, blog,
            input.Title, input.Slug,
            BlogPostStatus.Draft,
            input.ShortDescription, input.Content,
            input.CoverImageMediaId);

        input.MapExtraPropertiesTo(blogPost);
        await BlogPostRepository.InsertAsync(blogPost);

        return ObjectMapper.Map<BlogPost, BlogPostDto>(blogPost);
    }

    [Authorize(CmsKitAdminPermissions.BlogPosts.Update)]
    public virtual async Task<BlogPostDto> UpdateAsync(Guid id, UpdateBlogPostDto input)
    {
        var blogPost = await BlogPostRepository.GetAsync(id);
        blogPost.SetTitle(input.Title);
        blogPost.SetShortDescription(input.ShortDescription);
        // ... slug change goes through BlogPostManager.SetSlugUrlAsync
        // ... extra-property mapping
        // ... save and map
    }
}
```

Three patterns to notice:

**1. Layered attributes.** Every method inherits the class-level `[Authorize(CmsKitAdminPermissions.BlogPosts.Default)]` *plus* its own granular check (`Create`, `Update`, `Delete`). The class-level feature attributes mean the entire service compiles out when the global `BlogsFeature` is disabled.

**2. Manager + repository split.** The manager (`BlogPostManager.CreateAsync`) returns the constructed aggregate; the app service inserts it. Persistence is *not* the manager's job. This makes managers reusable in event handlers, background jobs, etc.

**3. `MapExtraPropertiesTo`.** ABP's `IHasExtraProperties` lets DTOs carry arbitrary extension data. `input.MapExtraPropertiesTo(blogPost)` copies all extension-method-defined fields from DTO to entity, which is how host apps add custom fields to `BlogPost` without modifying CMS Kit.

Note that the list DTO and detail DTO are *both* `PageDto` here — pages are small enough not to need a slimmed list shape.

```csharp
// PageAdminAppService.cs (excerpt)
[RequiresFeature(CmsKitFeatures.PageEnable)]
[RequiresGlobalFeature(typeof(PagesFeature))]
[Authorize(CmsKitAdminPermissions.Pages.Default)]
public class PageAdminAppService : CmsKitAdminAppServiceBase, IPageAdminAppService
{
    protected IPageRepository PageRepository { get; }
    protected ICommentRepository CommentRepository { get; }
    protected PageManager PageManager { get; }
    protected IDistributedCache<PageCacheItem> PageCache { get; }
    // ...

    public virtual async Task<PageDto> GetAsync(Guid id)
    {
        var page = await PageRepository.GetAsync(id);
        return ObjectMapper.Map<Page, PageDto>(page);
    }

    public virtual async Task<PagedResultDto<PageDto>> GetListAsync(GetPagesInputDto input)
    {
        var count = await PageRepository.GetCountAsync(input.Filter);
        var pages = await PageRepository.GetListAsync(
            input.Filter, input.MaxResultCount, input.SkipCount, input.Sorting);
        // ... map to PagedResultDto<PageDto>
    }
}
```

Page mutations also invalidate `PageCache` so the next public read hits the database — the cache is rebuilt on first miss.

## `IMediaDescriptorAdminAppService`

```csharp
public interface IMediaDescriptorAdminAppService : IApplicationService
{
    Task<MediaDescriptorDto> CreateAsync(string entityType, CreateMediaInputWithStream inputStream);
    Task DeleteAsync(Guid id);
}
```

`CreateMediaInputWithStream` carries an `IRemoteStreamContent` plus the original file name — this is how multipart uploads cross the ABP dynamic-proxy boundary. The implementation:

1. Calls `MediaDescriptorManager.CreateAsync(entityType, fileName, mimeType, length)`.
2. Inserts the resulting `MediaDescriptor` row.
3. Streams the bytes into `IBlobContainer<MediaContainer>` keyed by `descriptor.Id.ToString()`.

There is no `UpdateAsync` — to replace a file you create a new descriptor and delete the old one.

`MoveAllBlogPostsAsync` exists so an admin can decommission a blog without losing posts — bulk-reassign first, then delete the blog. If `assignToBlogId` is null, the posts are deleted with the blog (cascading to their comments via `BlogPostManager.DeleteAsync`).

## Base class and module wire-up

```csharp
// modules/cms-kit/src/Volo.CmsKit.Admin.Application/Volo/CmsKit/Admin/CmsKitAdminAppServiceBase.cs
public abstract class CmsKitAdminAppServiceBase : CmsKitAppServiceBase
{
    protected CmsKitAdminAppServiceBase()
    {
        ObjectMapperContext = typeof(CmsKitAdminApplicationModule);
    }

    /// <summary>
    /// Checks given policies until finding a granted one. If none is granted, throws <see cref="AbpAuthorizationException"/>.
    /// </summary>
    protected async Task CheckAnyOfPoliciesAsync([NotNull] IEnumerable<string> policies)
    {
        Check.NotNull(policies, nameof(policies));

        foreach (var policy in policies)
        {
            if (await AuthorizationService.IsGrantedAsync(policy))
            {
                return;
            }
        }

        throw new AbpAuthorizationException();
    }
}
```

`CmsKitAppServiceBase` (in `Volo.CmsKit.Common.Application`) is itself the project's `ApplicationService` subclass; it sets `LocalizationResource = typeof(CmsKitResource)` once for all CMS Kit app services. The admin base only overrides `ObjectMapperContext` so admin DTO mappings resolve out of `CmsKitAdminApplicationModule`'s AutoMapper profile.

`CheckAnyOfPoliciesAsync` is the helper used by methods like `BlogPostAdminAppService` when an action is permitted under any of several permissions (e.g. "Update *or* Publish" both let you save a draft change).

Two of the admin services — `BlogPostAdminAppService` and `TagAdminAppService` — actually derive directly from `CmsKitAppServiceBase` (skipping the admin base) for historical reasons; the rest go through `CmsKitAdminAppServiceBase`. New services should prefer the admin base.

`CmsKitAdminApplicationModule` depends on:

- `CmsKitDomainModule` — managers and repositories.
- `CmsKitAdminApplicationContractsModule` — interfaces and DTOs.
- `AbpDddApplicationModule` — `ApplicationService` base, `ObjectMapper`.
- `AbpAutoMapperModule` — DTO ↔ entity mapping in `CmsKitAdminApplicationMappers`.
- `AbpFeatureManagementModule` — for `[RequiresFeature]`.

```csharp
    public static class BlogPosts { /* same pattern */ }
    public static class Blogs     { /* same pattern */ }
    public static class Comments  { /* same pattern */ }

```

Hosts grant these to roles via the standard permission-management UI.
