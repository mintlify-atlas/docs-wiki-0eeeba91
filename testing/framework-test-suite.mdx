`framework/test/` contains **84 directories**: 81 test projects, two shared support projects (`AbpTestBase`, `Volo.Abp.TestApp`), and the `SimpleConsoleDemo` sample. Every directory maps one-to-one to a `framework/src/Volo.Abp.{Name}/` package — if there's a source project, there's almost always a `{Name}.Tests` next to it. This page enumerates them by category so you can find the prior art that matches whatever you're testing.

```
framework/test/
├── AbpTestBase/                                 // shared helpers
├── SimpleConsoleDemo/                           // sample, not a test project
├── Volo.Abp.TestApp/                            // shared "TestApp" entities/services
├── Volo.Abp.TestApp.Tests/                      // tests using TestApp + Memory DB
└── Volo.Abp.{...}.Tests/                        // 80 individual test projects
```

Every test project uses xUnit + Shouldly + NSubstitute. `common.test.props` (imported at the top of every `*.Tests.csproj`) provides the package references:

```xml
<!-- Excerpt from framework/test/AbpTestBase/AbpTestBase.csproj -->
<ItemGroup>
  <PackageReference Include="Microsoft.NET.Test.Sdk" />
  <PackageReference Include="NSubstitute" />
  <PackageReference Include="Shouldly" />
  <PackageReference Include="xunit" />
  <PackageReference Include="xunit.extensibility.execution" />
  <PackageReference Include="xunit.runner.visualstudio" />
</ItemGroup>
```

## Reading a test project — the recipe

Open any `framework/test/Volo.Abp.{Name}.Tests/` and you'll find the same handful of files:

```
Volo.Abp.{Name}.Tests/
├── Volo.Abp.{Name}.Tests.csproj      // imports ../../common.test.props
├── Volo.Abp.{Name}.Tests.abppkg      // ABP package metadata, mostly empty
├── Properties/AssemblyInfo.cs        // declares assembly-level attributes
└── Volo/Abp/{Name}/
    ├── Abp{Name}TestModule.cs        // module + DependsOn graph
    ├── Abp{Name}TestBase.cs          // : AbpIntegratedTest<Abp{Name}TestModule>
    └── {Subject}_Tests.cs            // one file per subject
```

`Volo.Abp.Core.Tests` is the smallest concrete example. It does not even have a `TestBase`:

```
framework/test/Volo.Abp.Core.Tests/
├── Microsoft/Extensions/DependencyInjection/
│   ├── AbpConventionalDependencyInjectionExtensions_Tests.cs
│   └── DependencyInjection_Tests.cs
├── System/
│   ├── AbpTypeExtensions_Tests.cs
│   ├── Collections/Generic/AbpCollectionExtensions_Tests.cs
│   ├── Collections/Generic/AbpListExtensions_Tests.cs
│   ├── ComparableExtensions_Tests.cs
│   ├── DayOfWeekExtensions_Tests.cs
│   ├── Linq/PredicateBuilder_Tests.cs
│   ├── ObjectExtension_Test.cs
│   └── StringExtensions_Tests.cs
└── Volo.Abp.Core.Tests.csproj
```

The directory structure under the project root mirrors the namespace under `Volo.Abp.Core` — `StringExtensions_Tests` sits at `System/StringExtensions_Tests.cs` because the class it tests is `System.StringExtensions` (an ABP-provided extension class).

## The .abppkg / .abppkg.analyze.json files

Every test project includes a tiny `*.abppkg` file. It is used by the ABP CLI to discover packages in the solution and by the `Volo.Abp.Studio` tooling — not by the test runner. You can ignore it when writing tests, but expect to see it next to every `.csproj`.

## How tests use the TestApp rig

`Volo.Abp.TestApp` is a miniature application reused by several test projects. It contains:

- `Person` and `City` entities (the `Person` has a navigation to `City`).
- `PersonRepository` (a default repository surface) and `IPersonRepository` plus extension methods.
- `IPersonAppService` / `PersonAppService` to test application-service plumbing.
- `AbpTestAppDomainModule` and `AbpTestAppApplicationModule` for the two layers.

`Volo.Abp.TestApp.Tests` references it and `Volo.Abp.MemoryDb`, then composes a startup module to verify the Repository abstractions against an in-memory store. The same TestApp is reused by tests in `Volo.Abp.Data.Tests` for filter and `IUnitOfWork` scenarios.

## When you should add a new test project versus extend an existing one

<AccordionGroup>
  <Accordion title="Add a new test project">
    You added a new package under `framework/src/Volo.Abp.X/` and need integration coverage. Copy the closest existing test project's `.csproj`, add an empty `AbpXTestModule.cs` that depends on your new module + `AbpTestBaseModule` + `AbpAutofacModule`, and pattern-match the existing `*TestBase.cs` from a similar package.
  </Accordion>
  <Accordion title="Extend an existing test project">
    You changed a public type and want one more `[Fact]`. Add the file under the namespace folder that matches the type's namespace. No infrastructure changes needed; the existing test base will pick the new file up by xUnit discovery.
  </Accordion>
  <Accordion title="Add an abstract repository test under TestBase">
    Your change touches the `IRepository` contract and both EF Core and Mongo need to cover it. Add the test as an abstract class under `Volo.Abp.{Module}.TestBase`, then add a thin subclass in each of `*.EntityFrameworkCore.Tests` and `*.MongoDB.Tests` to bind it to that store. The Identity test base uses this exact pattern — see [Module test patterns](/testing/module-test-patterns).
  </Accordion>
</AccordionGroup>

<Tip>
The lack of a test project for a given source folder is itself signal. `Volo.Abp.AspNetCore.Mvc.UI.Theme.Basic`, for instance, has no test project — it's a runtime theme whose behavior is covered indirectly through the `Theme.Shared.Tests`. When you add a new package, ask whether your code needs its own project or whether it should hook into an existing one.
</Tip>
