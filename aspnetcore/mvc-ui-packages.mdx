`Volo.Abp.AspNetCore.Mvc.UI.Packages` is a thin packaging module: one `IBundleContributor` per popular front‑end library, plus the resource‑mapping data that tells the ABP CLI where to copy the files at build time. Combined with `AbpMvcLibsService` (in `Volo.Abp.AspNetCore.Mvc`) and the `abp install-libs` CLI command (in `Volo.Abp.Cli.Core`), it provides a complete "BYO npm package" experience for the MVC UI.

Module path: `framework/src/Volo.Abp.AspNetCore.Mvc.UI.Packages/Volo/Abp/AspNetCore/Mvc/UI/Packages/`.

## `AbpAspNetCoreMvcUiPackagesModule`

The module ships under a `[DependsOn(typeof(AbpAspNetCoreMvcUiBundlingAbstractionsModule))]` declaration and registers an embedded virtual file set so all the contributor classes can resolve their pre‑shipped JS shims (small adapter files) without copying anything to `wwwroot` at build time.

Every contributor class extends `BundleContributor` (from `Volo.Abp.AspNetCore.Mvc.UI.Bundling.Abstractions`) and adds files under `/libs/<library>/...` to the bundle context. For example `JQueryScriptContributor.ConfigureBundle` adds `/libs/jquery/jquery.js`. Other contributors `AddDependency<JQueryScriptContributor>()` so the bundler emits jQuery first.

## Where the files come from — `libs.config`

ABP does not ship the upstream `.js` / `.css` files inside the NuGet package. Instead each library has a mapping entry that tells the CLI to copy from `node_modules` to `wwwroot/libs`. The mappings live in two places:

| File | Where |
| --- | --- |
| `abp.resourcemapping.js` | At the root of any MVC UI host project — e.g. `modules/basic-theme/test/Volo.Abp.AspNetCore.Mvc.UI.Bootstrap.Demo/abp.resourcemapping.js`. |
| Resource mapping inside each `Volo.Abp.AspNetCore.Mvc.UI.Packages.*` NuGet | Aggregated by the CLI at install time. |

Example file:

```js
// abp.resourcemapping.js
module.exports = {
    aliases: {
        "@node_modules": "./node_modules",
        "@libs": "./wwwroot/libs"
    },
    clean: [
        "@libs"
    ],
    mappings: {
        "@node_modules/highlight.js/**/*.*": "@libs/highlight.js/"
    }
}
```

The CLI reads this file, plus mappings contributed by every depended ABP package, and copies the matching files into `wwwroot/libs/` — replacing the historical "Bower restore" step.

## `AbpInstallLibsService` (CLI side)

The CLI command lives at `framework/src/Volo.Abp.Cli.Core/Volo/Abp/Cli/Commands/InstallLibsCommand.cs`. It delegates to `IInstallLibsService` (interface in `framework/src/Volo.Abp.Cli.Core/Volo/Abp/Cli/LIbs/IInstallLibsService.cs`) whose default implementation is `InstallLibsService` (`framework/src/Volo.Abp.Cli.Core/Volo/Abp/Cli/LIbs/InstallLibsService.cs`).

The flow is:

<Steps>
  <Step title="Discover MVC UI projects">
    `InstallLibsService` scans the solution for projects that reference any `Volo.Abp.AspNetCore.Mvc.UI.*` package.
  </Step>
  <Step title="Run yarn / npm install">
    For each such project the CLI runs `yarn` (preferred) or `npm install` against the project's `package.json` to populate `node_modules/`.
  </Step>
  <Step title="Execute the mapper">
    The CLI then invokes `gulp` against `abp.resourcemapping.js`, which copies files according to the `mappings` dictionary.
  </Step>
  <Step title="Result">
    `wwwroot/libs/` is repopulated. The MVC bundle contributors expect files at `/libs/<name>/<file>`, which now exist on disk and can be served by `UseStaticFiles()`.
  </Step>
</Steps>

Invocation:

```bash
abp install-libs
```

This is part of every solution's first run instructions — without it, the bundle contributors point at missing files.

## `AbpMvcLibsService` — the "missing libs" error page

When the MVC UI starts up with `AbpMvcLibsOptions.CheckLibs = true`, the framework wires a startup middleware that probes for the existence of a few well‑known files (e.g. `wwwroot/libs/jquery/jquery.js`). If they are missing, browsers get a friendly error page instead of an HTTP 500.

```csharp
// framework/src/Volo.Abp.AspNetCore.Mvc/Volo/Abp/AspNetCore/Mvc/Libs/AbpMvcLibsService.cs
public class AbpMvcLibsService : IAbpMvcLibsService, ITransientDependency
{
    private Task<bool>? _checkLibsTask;

    public virtual void CheckLibs(ApplicationInitializationContext context)
    {
        var options = context.ServiceProvider.GetRequiredService<IOptions<AbpMvcLibsOptions>>().Value;
        if (options.CheckLibs)
        {
            var app = context.GetApplicationBuilderOrNull();
            if (app == null)
            {
                var logger = context.ServiceProvider.GetRequiredService<ILogger<AbpMvcLibsService>>();
                logger.LogWarning($"The {nameof(IApplicationBuilder)} is not available. The 'CheckLibs' feature is disabled!");
                return;
            }

            app.Use(async (httpContext, next) =>
            {
                if (!httpContext.Request.IsAjax() && !await CheckLibsAsyncOnceAsync(httpContext))
                {
                    var errorPage = new AbpMvcLibsErrorPage();
                    await errorPage.ExecuteAsync(httpContext);
                    return;
                }
                await next(httpContext);
            });
        }
    }
}
```

`AbpMvcLibsOptions`:

```csharp
public class AbpMvcLibsOptions
{
    public bool CheckLibs { get; set; }
}
```

`AbpMvcLibsErrorPage.cshtml` (sitting next to the service) renders a styled page that says "Client‑side libraries are not installed yet. Run `abp install-libs` to install them."
