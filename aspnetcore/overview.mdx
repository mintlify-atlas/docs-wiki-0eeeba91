This group documents every package under `framework/src/Volo.Abp.AspNetCore.*` plus the closely related `Volo.Abp.Swashbuckle` package. These modules glue the framework's cross‑cutting concerns — unit of work, multi‑tenancy, auditing, exception handling, authorization, dynamic claims — into the ASP.NET Core request pipeline. They are the layer where `AbpModule.OnApplicationInitialization(...)` calls `app.UseConfiguredEndpoints()`, `app.UseAbpExceptionHandling()`, `app.UseMultiTenancy()`, `app.UseUnitOfWork()` and `app.UseAuthorization()` from a Web host's startup module.

<Note>
  All file paths cited in this group are relative to `/home/daytona/repos/abpframework/abp/`. The host module is `framework/src/Volo.Abp.AspNetCore/Volo/Abp/AspNetCore/AbpAspNetCoreModule.cs`; the MVC module is `framework/src/Volo.Abp.AspNetCore.Mvc/Volo/Abp/AspNetCore/Mvc/AbpAspNetCoreMvcModule.cs`. See [/core/modularity](/core/modularity) for how `[DependsOn]` chains these together and [/flows/http-request-pipeline](/flows/http-request-pipeline) for an end‑to‑end request walkthrough.
</Note>

## The ABP request pipeline at a glance

Every Web host built on `Volo.Abp.AspNetCore` calls `OnApplicationInitialization(...)` in some `*HttpApiHostModule`. The standard ordering — established by the application templates and reproduced by the modules in this group — is shown below.

```mermaid
flowchart LR
    Req([HTTP request]) --> Forwarded[UseForwardedHeaders]
    Forwarded --> Static[UseStaticFiles]
    Static --> Cors[UseAbpRequestLocalization]
    Cors --> Excpt[UseAbpExceptionHandling]
    Excpt --> Route[UseRouting]
    Route --> Auth[UseAuthentication]
    Auth --> Tenant[UseMultiTenancy]
    Tenant --> Authz[UseAbpClaimsMap / UseAuthorization]
    Authz --> Serilog[UseAbpSerilogEnrichers]
    Serilog --> Uow[UseUnitOfWork]
    Uow --> Endpoints[UseConfiguredEndpoints]
    Endpoints --> Controller[[MVC controller / Razor page]]
    Controller --> Filters[Action filters: GlobalFeature, Validation, Audit, ExceptionFilter, Uow]
    Filters --> Service[[Application service]]
    Service --> Resp([HTTP response])
```

The middleware names map directly to source files:

| Middleware | Type | Path |
| --- | --- | --- |
| `UseAbpExceptionHandling` | `AbpExceptionHandlingMiddleware` | `framework/src/Volo.Abp.AspNetCore/Volo/Abp/AspNetCore/ExceptionHandling/` |
| `UseMultiTenancy` | `MultiTenancyMiddleware` | `framework/src/Volo.Abp.AspNetCore.MultiTenancy/.../MultiTenancyMiddleware.cs` |
| `UseUnitOfWork` | `AbpUnitOfWorkMiddleware` | `framework/src/Volo.Abp.AspNetCore/Volo/Abp/AspNetCore/Uow/` |
| `UseAbpSerilogEnrichers` | `AbpSerilogMiddleware` | `framework/src/Volo.Abp.AspNetCore.Serilog/.../AbpSerilogMiddleware.cs` |

Inside MVC, `AbpMvcOptionsExtensions.AddAbp(MvcOptions, IServiceCollection)` registers the action filters (`GlobalFeatureActionFilter`, `AbpValidationActionFilter`, `AbpAuditActionFilter`, `AbpExceptionFilter`, `AbpUowActionFilter`) via `options.Filters.AddService(typeof(...))` — see `framework/src/Volo.Abp.AspNetCore.Mvc/Volo/Abp/AspNetCore/Mvc/AbpMvcOptionsExtensions.cs`.

## What sits where in the source tree

The packages in this group all live under `framework/src/`. The naming follows a strict convention:

| Folder | Module type |
| --- | --- |
| `Volo.Abp.AspNetCore/` | The host module. Every other Web module depends on it. |
| `Volo.Abp.AspNetCore.Abstractions/` | Public contracts. Used by packages that mustn't take the full Web host dependency. |
| `Volo.Abp.AspNetCore.Mvc/` | The MVC integration — easily the largest package in the group, ~150 source files. |
| `Volo.Abp.AspNetCore.Mvc.Contracts/` | DTOs and interfaces shared between the MVC module and its HTTP API clients (`ApplicationConfigurationDto` lives here). |
| `Volo.Abp.AspNetCore.MultiTenancy/` | `MultiTenancyMiddleware` plus the six HTTP tenant resolvers. |
| `Volo.Abp.AspNetCore.Mvc.UI.MultiTenancy/` | The tenant switcher UI dropdown for MVC themes. |
| `Volo.Abp.AspNetCore.Serilog/` | One middleware that pushes ABP identity onto `LogContext`. |
| `Volo.Abp.AspNetCore.SignalR/` | `AbpHub` base, `AbpSignalROptions`, auto‑hub‑mapping. |
| `Volo.Abp.AspNetCore.Authentication.JwtBearer/` | Dynamic claims for stateless JWT‑secured APIs. |
| `Volo.Abp.AspNetCore.Authentication.OAuth/` | `MultipleClaimAction` / `RemoveDuplicateClaimAction` helpers. |
| `Volo.Abp.AspNetCore.Authentication.OpenIdConnect/` | Cookie + OIDC flow with local user creation. |
| `Volo.Abp.AspNetCore.Bundling/` and `.Mvc.UI.Bundling*` | Bundle abstractions and MVC implementation. |
| `Volo.Abp.AspNetCore.Mvc.UI.Widgets/` | View‑components‑as‑widgets. |
| `Volo.Abp.AspNetCore.Mvc.UI.Packages/` | Standard bundle contributors for jQuery, Bootstrap, DataTables, etc. |
| `Volo.Abp.AspNetCore.TestBase/` | `WebApplicationFactory`‑based integration test base. |
| `Volo.Abp.Swashbuckle/` | ABP‑aware Swagger UI and OpenAPI document filters. |

## Conventions every package in this group follows

<Steps>
  <Step title="One AbpModule per assembly">
    Each `csproj` ships exactly one class named `Abp{Suffix}Module : AbpModule`. It declares `[DependsOn(...)]` against the modules it logically depends on — `AbpAspNetCoreSignalRModule` depends only on `AbpAspNetCoreModule`, for example.
  </Step>
  <Step title="ConfigureServices / OnApplicationInitialization">
    Modules register options and DI services in `ConfigureServices`, and wire up middleware via `AbpEndpointRouterOptions` or the `app.UseXxx()` extension method exposed under the `Microsoft.AspNetCore.Builder` namespace.
  </Step>
  <Step title="Middlewares extend AbpMiddlewareBase">
    `AbpMiddlewareBase` declares `Task InvokeAsync(HttpContext, RequestDelegate)`. The DI conventional registrar registers every derived class as `ITransientDependency`, so `app.UseMiddleware<TMiddleware>()` resolves it from the request scope.
  </Step>
  <Step title="Options classes are plain POCOs">
    Every `Abp*Options` class is a POCO with init‑only collections and public setters for primitives. They are configured via `Configure<TOptions>(action)` from inside `ConfigureServices`, with `PreConfigure<TOptions>(...)` used when the module itself reads them in `PreConfigureServices`.
  </Step>
</Steps>
